## Changed

- **README brought back in line with the app, and given the new artwork.** It was advertising **rack presets**, removed in v2.12.0, and showing `Panda_Feed_Rack_Standard.png`, deleted with them — a dead image on the GitHub page. The presets bullet now describes the actual rack builder, and the image points at the artwork that still exists. The new ecosystem render opens the intro *above* the founder's real-bench photo, which stays: the render shows what the software is, the photo proves someone actually uses it. The TigerPOD line-up banner illustrates its section — nine printed colours say "free STL, print it your colour" better than the sentence under it.
- **`docs:check` now validates image paths too** (`src="…"` and `![](…)` in README / FEATURES / ROADMAP). It only checked paths written in backticks, which is why a deleted illustration sat broken in the README without anything noticing — `scripts/check-docs-drift.mjs`
- **`README.md` joined the release checklist** (`CLAUDE.md`). It was the only user-facing doc not in it, so it silently lagged — v2.13.0 shipped container calibration without the README ever mentioning it (added now). Nothing breaks when the README is stale, which is exactly why it gets forgotten, and `docs:check` cannot catch it: completeness of prose is not a derivable fact. The step names the litmus test — *would someone deciding whether to download the app want to know this exists?* — so only real `## Added` capabilities go in, not fixes or internal work — `CLAUDE.md`, `README.md`

- **README now states what this project actually is** — a new *"A sandbox, not the product"* section, placed before the ecosystem table. Tiger Studio is one piece of TigerSystem and deliberately a laboratory: a demonstration of what open, documented technology allows once a spool identifies itself, neutral and agnostic between filament brands, printer makers and distributors — not a walled garden with a partner list. It closes by telling the reader to build their own: TigerTag is an open protocol, not a platform to join, so nothing here is a prerequisite — read the chip format, take an SDK, build a different ecosystem, it will still speak the same chips — `README.md`
- **`llms.txt` rewritten to match the app, and made self-guarding.** It is the file an AI reads FIRST, so its drift is the most expensive kind here — and it had a lot. Its Firestore map documented 15 collections while the renderer writes 24: missing were `products` (the identity carrying buy links, prices and attachments), `lists`, `publicLists`, `containerOverrides`, `rfidList`, `notifications`, `apps/studio`, `printers/{brand}/devices` and `printerSeen`. Added on top: the project's positioning as an agnostic laboratory on an open protocol; a **spool-tiers** section defining TigerData / TigerTag / TigerTag+ *and* recording that the TigerCloud→TigerData rename is display-only with no migration, so no future session renames stored values for cosmetics; `tier` as a server-only field gating the attachment cap; which collections friends can read; and that the rules live in another repo and are field-whitelisted (unlisted fields fail silently) — `llms.txt`
- **`docs:check` now proves the Firestore map is complete** — it extracts every literal `.collection("…")` the renderer writes and fails the commit on any collection absent from `llms.txt`. Written because the six-collection gap above accumulated over several releases with nothing to notice it; on its first run it immediately found three more (`devices`, `printerSeen`, `publicLists`) that a hand-written grep had missed — which is the whole argument for deriving the list from the code instead of trusting a human list. Verified it blocks by adding a fake collection — `scripts/check-docs-drift.mjs`
- **`llms.txt` joined the release checklist** as its own step, next to `README.md` (`CLAUDE.md`). The validator guards what is derivable; a new subsystem or a renamed user-facing term is prose and cannot be. The step names the litmus test — *would an agent starting cold get this wrong by reading only llms.txt?*
- **Standing review brief** — `docs/REVIEW-BRIEF.md`, a read-only audit brief covering security, bugs, performance, first-try UI correctness, development velocity / token cost, and the brand-licence-certification foundations behind selling chips to manufacturers. Kept in the repo so a full review starts with one line instead of a long pasted prompt — `docs/REVIEW-BRIEF.md`
