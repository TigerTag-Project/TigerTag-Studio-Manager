# Worklog — v2.13.0 (in progress)

## Added

- **`.circle-center` — a utility for content optically centred in a round badge** (`renderer/css/00-base.css`), for step badges, counters, initials, icon discs. The usual `flex + align-items:center` drifts on numerals: it centres the *line box*, whose height includes the font's ascender and descender, and a digit has no descender — so it lands slightly high; an inherited line-height makes it worse, and monospace faces worst of all (their vertical metrics are built for column alignment, not optical centring). The utility uses `place-items:center` on a grid (centres the item box, not the line box), `line-height:1` to strip the leading, `tabular-nums` so a counter never widens between 1 and 11, and a `.06em` top nudge for the half-descender digits never use. Applied to the calibration modal's three "How to measure" badges and the rack side-card's three step badges, which also dropped the mono face that skewed them — `renderer/css/00-base.css`, `renderer/css/60-modals.css`, `renderer/inventory.html`

- **`npm run docs:check` — a validator for the docs that restate the code**, wired into the pre-commit hook next to `i18n:check` and `codemap:check`. Several docs repeat numbers the code owns (version, renderer line count, printer-brand count, i18n key count, folder paths); nothing breaks when they drift, so nobody notices — `llms.txt`, the file an agent reads first, had reached **v1.8.2 / ~12 000 lines / 5 brands** against a real v2.12.0 / 28 500 / 6, with Anycubic absent from its protocol table. The check compares each stated fact to its source of truth (with tolerances where the number moves constantly), flags doc references to paths that no longer exist, and skips templates and globs. It only checks derivable facts — never prose. Runs when a doc, or a file it describes, is staged — `scripts/check-docs-drift.mjs`, `.githooks/pre-commit`, `package.json`, `CLAUDE.md`

- **`assets-src/` — one home for every full-resolution image that must never be bundled.** Versioned but outside the build (`build.files` is an allowlist and doesn't list it), mirroring each asset's path under `assets/` so finding a master is mechanical. Optimising an asset in place destroys its original; this gives it somewhere to live first, so a later re-export (bigger display size, different crop, retina variant) starts from the master rather than an already-compressed file. **`brand/` was folded into it** (its two TigerPOD masters moved to `assets-src/img/`, the folder removed): both existed for the same reason — HD, not shipped — and the "does the app load it?" distinction between them wasn't worth a second folder to look in — `assets-src/README.md`

- **Container weights can be corrected, per account.** The app ships a container catalogue (`data/container_spool/spools_filament.json`) whose empty weights are manufacturer values; a user who actually weighs theirs can now fix the number. A pencil sits to the LEFT of the weight in the "Choose a container" side-card (discreet until the row is hovered) and opens a **calibration modal** (`#cwModalOverlay`) framed as calibrating the container rather than editing a field: a large photo of the container beside its identity, the value as a grouped `[− value +]` stepper with the unit inside, a "a precise scale gives the best result" note (new `icon_scale.svg` — the existing `icon_tigerscale_3d` is a detailed product visual that flattens to an unreadable blob as a monochrome mask, and named the device rather than the action), and a **catalogue-vs-yours comparison** — two columns separated by an arrow in a white disc, their figures bottom-aligned (`margin-top: auto`) so they stay on one line however many lines each subtitle takes — whose right-hand figure follows the field as you type — so the delta is visible before saving, and a "yours" badge appears only once the value actually departs from the catalogue. The picker row carries the same badge, and a quiet "Reset to catalogue" sits opposite Save.
  - Stored in a new **`users/{uid}/containerOverrides/{containerId}`** doc (`{ containerWeight, updatedAt }`, owner-only rule added and deployed) and applied over the bundled JSON at read time — **the JSON is never modified**.
  - Resolution happens in `containerFind()`, the single accessor every consumer already goes through, so the corrected weight reaches the picker, the detail panel and future spool writes at once.
  - An **amber alert strip** sits just above the save choice and names how many spools currently inherit this weight, so the stakes are stated right where the decision is taken. It counts *usage*, not impact — it explains why the value matters even when nothing would change today (new `icon_alert.svg`).
  - Because each spool stores a *snapshot* of the weight it was given, a correction does not reach them on its own. A **"When saving" choice** sits last, just above the footer and after the catalogue-vs-yours comparison — you decide once you have seen the delta. It appears whenever the container is in use, decided **once on open** rather than live against the typed value — a live gate made the card appear and disappear mid-typing, resizing the modal under the cursor, and a choice that is occasionally a no-op costs less than that. Applying only writes to the spools whose weight actually differs, so the no-op case is free — "Update all N spools with this value" (default: the correction actually takes effect) or "Keep current spool values" — naming how many spools are concerned, and shown only when there are any. One decision, taken before committing, rather than a second screen after the fact; the alignment runs after the dialog closes, and is twin-mirrored so both chips of one physical spool stay in agreement.
  - A **"How to measure" strip** shows what goes on the scale as a left-to-right equation: the container's own photo **+** the cardboard core **=** the scale. Each card carries its number and label on one top line, and the operators are plain white discs — no border, no shadow — sitting as flex items BETWEEN two cards rather than positioned against one of them, so they are centred in the gap by construction. `margin: calc(-1 * (var(--op-size) + var(--how-gap)) / 2)` makes a disc's net flow width exactly `-gap`, so the cards keep their intended spacing while the disc's centre lands on the gap's midpoint (verified at 0.0000 px offset). Three earlier attempts anchored the disc to a card edge and kept drifting; deriving the geometry instead of eyeballing it settled it. The core card and its trailing joint hide together (two operators between two cards would be nonsense) and the remaining joint flips from "+" to "=", so the equation stays true with the middle term removed. The middle term drops out for containers that have no core, and the last step renumbers itself — a Masterspool is a reusable shell whose core stays behind with the spool, every other type (plastic / cardboard / customizable) is just the empty spool. Driven off the catalogue's `type` field. The pictures use one uniform height with `width: auto` rather than a square box: `aspect-ratio: 1` reserved vertical space no subject used — the scale is landscape, so a 112 px box held a 65 px subject and left 47 px of dead air under it. The two assets we control are exported cropped to their subject on an **alpha threshold of ~24, not 1**: a render's diffuse shadow carries a couple of alpha units across most of the canvas, so a threshold-1 trim kept it all — the scale was a 1492×1023 frame around a 1017×590 subject, which is where the empty space came from. Cropped properly, the subject now fills 91-95 % of its frame and the box hugs it; the spool comes from the catalogue on a square canvas with its own margin, and catalogue art is shared with every other view so it is not re-cropped. Each image therefore gets its own box height, tuned so the three SUBJECTS render at the same ~67 px — which is what the eye compares — rather than the same box. The core picture is `oem_carton_core.png` (trimmed + resized 1080² → 320² and palettised, **1.17 MB → 37 KB**); the scale is `oem_kitchen_scale.png` (cropped and palettised 1536×1024 → 361×220, **2.1 MB → 13 KB**) — squared on purpose, since the three steps render in equal tiles and an untrimmed 884×631 subject would have appeared smaller than its neighbours. The masked-glyph fallback stays wired in case the file ever goes missing.
  - The picker rows became `div[role=button]` (a `<button>` cannot legally contain the edit `<button>`), with Enter/Space selection restored.
  — `renderer/inventory.js`, `renderer/inventory.html`, `renderer/css/70-detail-misc.css`, `renderer/css/60-modals.css`, backend `firestore.rules`. i18n: `cpEditWeight`, `cpWeightEdited`, `cpResetWeight`, `cpApplyAsk`, `cpApplyYes`, `cpApplyNo`, `cwModalTitle`, `cwModalSub`, `cwWeightLabel`, `cwScaleHint`, `cwCatalogueLabel`, `cwCatalogueSub`, `cwYoursLabel`, `cwYoursSub`, `cwSaveBtn`, `cwResetBtn`, `cwWhenSaving`, `cwUsedBy`, `cwLinkedSpools`, `cwApplyAll`, `cwKeepCurrent`, `cwHowTitle`, `cwStepSpool`, `cwStepCore`, `cwStepScale`, `cwStepScaleSolo`, `cwLinkedSpools` — 9 locales.

## Changed

- **No dimming scrim behind the "Add product" side-card**, matching the rack editor. Both are editors acting on the inventory behind them, not blocking dialogs, so that inventory stays readable. The overlay stays in the DOM and keeps its `pointer-events`, so click-outside still closes the panel — only the black wash is dropped — `renderer/css/60-modals.css`

- **The "Stock value" tile now says whether it is excl. or incl. tax.** The figure already followed the account's HT/TTC preference but never named it, so `236.60 €` was ambiguous — and the two readings differ by the VAT rate. The qualifier reuses the same `reorderHT` / `reorderTTC` wording as every other price in the app, appended to the tile's `label` in a quieter weight (it qualifies the figure, it does not name the tile). It must go on `label`, which is rendered as HTML — putting it on `mini` broke the card, because `mini` is piped into a `data-mini` attribute for the collapsed sidebar and the markup's quotes closed the attribute early. That attribute is now escaped so the mistake cannot repeat. No extra wiring needed: `saveAccountPriceMode()` already calls `renderStats()`, so switching mode repaints it — `renderer/inventory.js`, `renderer/css/00-base.css`

- **The inventory's "Protocol" filter is now labelled "Type"** in all 9 locales (`filterShortProtocol`, plus the hardcoded HTML fallback). The key name is unchanged — it still filters the `protocol` field, and the label is display only, same split as the TigerData rename — `renderer/locales/*`, `renderer/inventory.html`

- **The three tier badges reworked — order, geometry and colour.** Tiles now follow the capability ladder, **TigerData → TigerTag → TigerTag+** (no chip → chip → chip + online layer). The three share one capsule — 18 px tall, fully rounded, 1 px hairline border, 10 px/650 type — and one construction borrowed from the wishlist buy button (`.lv-buy`): a top-lit vertical gradient, a matching border and dark type. Only the metal changes:
  - An early pass gave both chips a faux-metal treatment (horizontal five-stop gradient + text-shadow) and it clashed: it simulated a bar of metal in an otherwise flat UI, and at 10 px read as dull beige and muddy yellow. The badges ended up reusing two recipes the app already had rather than inventing a third.
  - **TigerTag+ borrows the wishlist buy button's exact colours** (`.lv-buy`): the same amber border, the same top-lit vertical gradient and the same dark brown type — one golden-yellow across the app meaning "this is the good one". It reads 8.3–9.7:1, and its slight vertical sheen is what separates the top of the ladder from the two flat tiers below.
  - **TigerTag is the silver counterpart**, built exactly like TigerTag+ — same top-lit vertical gradient, same border-plus-dark-type structure, amber swapped for silver (8.2–10.1:1). The two real chips therefore read as one pair, a gold rung and a silver rung.
  - **TigerData is white metal**, one step lighter again, so the three form a continuous ladder — white → silver → gold — all built the same way. Its border is a touch stronger than the other two: on a white card the capsule would otherwise have no edge to sit on (7.7:1 / 6.7:1).
  - **TigerTag** was `--surface-2` on `--muted`: a grey pill on a grey card, effectively invisible. It is now steel with white type (4.53:1). Moving it and TigerTag+ off orange also frees the app's ACTION colour — a tier badge wearing the same orange as every button competed with every call to action.
  - **TigerData** dropped its purple, which was picked years ago only to "differ from orange" and occupied the last free slot in an app where orange means action, green success, red danger and sky-blue twin-pairing. It is now the lightest capsule of the three — the visual bottom of the ladder, and the only one that is not a chip you can buy.
  - A real `border` is used rather than an inset shadow: `box-sizing: border-box` is global and the capsule height is fixed, so the border costs no outer size. All three contrasts clear **WCAG AA**.
  - The stat-tile override in `00-base.css` used to shrink these badges (9 px / 1 px 6 px); that fought the fixed-height capsule and was reduced to the `text-transform` reset it actually needs, so a badge looks the same in the header and on a spool card.
  - `.tag-twin` (twin pairing, sky-blue) still uses a gradient — deliberately out of scope, but it is now the last badge that does — `renderer/inventory.js`, `renderer/css/70-detail-misc.css`, `renderer/css/00-base.css`

- **"TigerCloud" is now displayed as "TigerData".** A display-only rebrand of the fully-digital tier: the tier badge on a spool, the header stat tile, the encode-flow title ("TigerData → TigerTag"), and the `statCloud` / `productCloudCreated` strings across all 9 locales (the brand name itself isn't translated; the surrounding sentence stays in each language). Forward-facing docs follow — README, `FEATURES.md`, `ROADMAP.md`.
  - **The stored value stays `TigerCloud` everywhere it is data**, untouched: the Firestore `protocol` field written by `normalizeRow` and `syncSpoolMirrors`, the stat tile's `filter: "TigerCloud"` (compared against that field), every filter comparison, and the backend's `byProtocol` map key. Renaming any of those would silently orphan the tier for every existing spool of every user, so `docs/firestore-schema.md` now carries an explicit warning next to the field, and the code comments that document the stored value were deliberately left as-is.
  - The **What's New history was rewritten too** (29 strings across 8 past versions): that modal is browsable in-app via its version picker, so it is live UI, not an archive — showing a tier name that no longer exists anywhere in the app would be more confusing than restating it. The frozen records that are NOT re-openable in the app keep the original wording: `CHANGELOG.md`, `data/release-notes/*` (already published to GitHub Releases — editing them locally changes nothing there) and `DEVLOG.md`.
  - **Tiger Hub follows** (separate repo, same discipline): 9 locales, the account + admin stat panels, press grid, tier ladder, `BRIEF.md` naming law, README/MAP/CLAUDE/llms route. Two Hub spots rendered the raw stored value to the user — the admin protocol bars (the chart prints its lookup key as the label) and the stock badge — and now map key→label so the lookup keeps matching the rollup.
  - The **protocol quick-filter** next to the search bar still read "TigerCloud": `populateOneQuickFilter` used each stored value as both the option's `value` and its visible text. It gained an optional `labelOf` projection — the value stays the stored string the filter compares against, only the text changes — so the dropdown (and the custom popup, which mirrors `option.text`) now reads "TigerData" while still selecting the right spools.
  - A **third, older variant — "TigerTag Cloud" (with a space)** — was still alive in five `inventory.js` comments and in `FEATURES.md`; both now read "TigerData". The FEATURES line was doubly stale: it described a "Create a TigerTag Cloud" action that has been labelled "+ Material" since v2.2.0, so it now names the real button. Untouched elsewhere: `TigerTag_Firebase_Integration/rules/storage.rules` says "TigerTag Cloud Storage Security Rules" — that is Firebase **Cloud Storage**, the Google product, not this tier.
  - No data migration, no backfill, and the backend repo is untouched — `renderer/inventory.js`, `renderer/inventory.html`, 9 locales, `data/whatsnew.json`, `README.md`, `FEATURES.md`, `ROADMAP.md`, `docs/firestore-schema.md`

- **Ecosystem cross-links + `llms.txt` refresh.** `README.md` now points TigerPOD at its own repository (it only linked the MakerWorld STL), in both the ecosystem table and its dedicated section. `llms.txt` had drifted a long way from reality — version `1.8.2` → `2.12.0`, renderer "~12 000 lines" → ~28 500, "5 brands" → 6 (Anycubic was missing from the protocol table entirely), 8 CSS files → 10, i18n "791 keys as of v1.8.2" → 1346 as of v2.12.0 — and it gained an **Ecosystem repositories** table (both SDKs, TigerScale, TigerPOD) plus `FEATURES.md` / `CODEMAP-main.md` in its docs list — `README.md`, `llms.txt`

## Fixed

- **The custom-colour swatch's pencil vanished on a white or pale colour** in the Add-product picker. The icon was hardcoded white with a double drop-shadow halo trying to keep it legible on any background — a workaround that failed on light colours and read as a smudge on the rest. It now flips black/white against the swatch via `readableTextOn()`, the same relative-luminance helper the avatars use, set inline at render time; the halo is gone. The CSS `background: #fff` stays as the fallback for a swatch the helper cannot parse. Only this renderer builds that cell, so nothing else changes — `renderer/inventory.js`, `renderer/css/60-modals.css`

- **Stepper − / + buttons didn't fill their control.** The global `button { height: 40px }` (70-detail-misc.css) is more than a default here: neither `.cw-step` nor `.rec-step` set a height, so both sat top-aligned inside a taller wrapper with a hover fill that stopped short of the edges — visible on the container-calibration stepper (66 px tall) and latent on the rack side-card's. Both now use `height: auto` + `align-self: stretch` so they follow the wrapper — `renderer/css/60-modals.css`
