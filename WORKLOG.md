# Worklog — v1.10.29 (in progress)

## Added
- Detail panel: new "Product ID" row under the UID, shown only for TigerTag+ — the on-chip `id_product` that ties the chip to its catalog entry (hidden when unset: 0 / 0xFFFFFFFF). The value is a link that opens the product page in the browser (`https://tigertag.io/pages/product-infos/<id>`, via an `infoRows` optional href + `target="_blank"` → `shell.openExternal`) — `renderer/inventory.js` (detail `infoRows` + render), `renderer/css/70-detail-misc.css` (`.pv-link`), i18n `detProductId`
- In-app paxx release notification — when at least one Snapmaker in the device list RUNS a paxx build older than the latest release, a clickable notice appears in the notification center, titled with the machine's configured name ("New firmware for U1-showroom"). Three deliberate actions: (1) clicking the CARD jumps to the machine — opening its detail side-card AND the Printer Settings side-card, where the firmware download CTA + status line live (`printerId`/`printerName` stored on the notice) — AND hides it for the session; (2) a "Remind me later" button (orange primary + bell icon) hides it for the session only; both are an in-memory, tag-scoped session flag (`_paxxSessionSnoozeTag`) so the notice comes back on the next app start (while the machine is still out of date), and a brand-new release breaks through immediately; (3) a ✕ close button is the ONLY permanent block — this exact release never notifies again (localStorage `tigertag.paxx.dismissedTag`, tag-scoped so a newer release still notifies). Both buttons stopPropagation so they don't trigger the card's open. The installed build is read from the machine itself (`paxxProbeInstalled` → `GET /firmware-config/api/status`, an endpoint that only exists on paxx firmware — stock machines are never nagged; path added to the `snap:http-get` IPC allowlist). Probes throttled to once per hour per session — `renderer/printers/snapmaker/paxx.js` (`paxxProbeInstalled`/`paxxBuildOf`/`paxxUpdateAvailable`), `renderer/inventory.js` (`_maybeNotifyPaxxRelease`, `_openPaxxPrinter`, `_snoozePaxxNotif`, `_dismissPaxxNotif`), `renderer/css/20-friends.css` (`.notif-later`, `.notif-dismiss-paxx`), `main.js`, i18n `notifPaxxTitle`/`notifPaxxText`/`notifRemindLater`
- Snapmaker settings side-card: installed-firmware status line under the paxx download CTAs when the machine's IP is known. User-facing, localized copy that states the action plainly instead of dumping the raw build string — "A newer firmware is available — download it just above" (accent) or "This printer is on the latest firmware. Nothing to do." (muted); the clean build (commit hash trimmed via `paxxCleanVersion`) stays available on hover (`title`). Silent for stock/unreachable machines — `renderer/printers/snapmaker/index.js`, `renderer/css/50-snapmaker.css` (`.snap-paxx-status`), i18n `snapPaxxUpdate`/`snapPaxxUpToDate`
- Printer-brand notifications speak with the brand logo — a notification center notice originating from a printer now shows the tinted brand logo (the machine "talking"), the same way a friend notice shows the friend's avatar. Generic: any local notif carrying a `brand` field renders the masked `logo_<brand>.svg` in a round chip (reuses `.pt-thumb-logo[data-brand]`). Applied to the paxx firmware notice (`brand: "snapmaker"`) — `renderer/inventory.js` (`renderNotifCenter`), `renderer/css/20-friends.css` (`.notif-ic--brandlogo`)
- New-notification chime — a soft two-note rising sound (synthesized via Web Audio, no asset) plays when a notification ARRIVES: an incoming friend request, a friend accepting yours (general notif), or a local notice (paxx firmware, app update). Rings on the real arrival event — Firestore `docChanges()` of type "added" AFTER each source's first snapshot (so the history that loads on login stays silent), and a freshly-pushed local notice — never on a plain re-render or a removal. Passive startup nudges (avatar / community links) don't ring — `renderer/inventory.js` (`_notifChime`, `_chimeOnFirestoreArrival`, `_notifFirstSnap`, `_PASSIVE_NOTIF_IDS`; wired into `subscribeFriendRequests`, `subscribeNotifications`, `_setLocalNotif`)
- Paxx firmware watch — daily GitHub Action that opens an issue in this repo when the Paxx U1 Extended Firmware publishes a new release (tag, download link, size, release notes; idempotent via issue-title dedup, no state file, `firmware-watch` label) and auto-closes the superseded ones, so the single open issue always means "latest available". Human-notification channel only — the app resolves the latest release itself via `paxx.js` — `.github/workflows/paxx-firmware-watch.yml`
- Toolbox info bubbles — every toolbox row now shows an ⓘ (in place of the decorative chevron) that reveals a plain-language explanation bubble on hover, instead of a native tooltip. The bubble is body-appended and positioned in JS (`#toolInfoPop`, flips above/below, viewport-clamped), so it escapes the rows' `overflow:hidden` and works on every row type (default + split); a delegated capture handler swallows pointer/click on the ⓘ so pressing it never fires the row. Every tool carries an `info` field — the 4 chip actions reuse their tips; 8 new benefit-first tips were written for Duplicate / Measure-color / Measure-TD / Edit-image / Twin-link / Remove-from-rack / Encode-cloud / Burn-RFID / Refresh-API (Duplicate + Twin-unlink tips also de-jargoned) — `renderer/inventory.js` (toolbox render, `_wireToolInfoTips`/`_showToolInfoTip`, `info` on all tools; called from `openDetail`), `renderer/css/70-detail-misc.css` (`.tool-info` / `#toolInfoPop`), i18n `toolMeasureColorTip`/`toolMeasureTdTip`/`toolEditImgTip`/`twinLinkActionTip`/`toolRemoveFromRackTip`/`encodeCloudTip`/`burnRfidTip`/`toolRefreshApiTip`
- Recycle to NFC — a toolbox action next to Reset that writes the SDK's blank-NDEF payload (`TigerTag.erase()` — zeroed user memory) so the chip leaves the TigerTag format entirely and becomes a plain generic NFC tag, then removes the material (+ twin) from the inventory (rfidList backup kept). User-facing label "Recycle to NFC" (the "generic NDEF format" detail lives in its ⓘ bubble). Shares the same present-chip targeting, UID guard, surgical write + verify, and hold-to-confirm as Reset (both go through the shared `_writeChipImage()` in main + `_wireChipReset()` in the renderer). New `icon-recycle` (`icon_recycle.svg`, Lucide recycling symbol) — `main.js` (`rfid:erase`), `preload.js` (`eraseRfidTag`), `renderer/inventory.js` (toolbox tool `btnEraseRfid`), `renderer/css/70-detail-misc.css`, i18n `toolEraseRfid`/`toolEraseRfid2`/`toolEraseRfidTip`/`toolEraseWriting`/`toolEraseDone`/`toolEraseFailed`
- Erase TigerTag / TigerTag+ — a toolbox action that reinitializes this spool's present chip(s) using the official SDK Init payload (`TigerTag.asInit()`: id_tigertag = TIGER_TAG_INIT, id_product = 0, fresh timestamp) written over pages 0x04-0x17, with the rest of the user region (0x18-0x27, any residual TigerTag+ signature) zeroed → a clean 144-byte Init. This is how a TigerTag+ gets a second life as a reusable plain TigerTag; the ⓘ bubble explains it. An Init chip carries no material/brand, so it doesn't parse as a spool and re-reading it never resurrects the previous one. The label is tier-aware ("Erase the TigerTag" / "Erase the TigerTag+", `{{tier}}` param). Then the material (+ twin) is removed from the inventory; the rfidList backup is kept, so a Restore stays possible for that UID. Surgical + read-back verify, UID-guarded per target. Shown only when at least one of this spool's chips is on a reader; hold-to-confirm — `main.js` (`rfid:format` via `TigerTag.asInit`), `preload.js` (`formatRfidTag`), `renderer/inventory.js` (toolbox tool `btnFormatRfid` + wiring), reuses `icon-broom`, i18n `toolFormatRfid`/`toolFormatRfid2`/`toolFormatRfidTip`/`toolFormatWriting`/`toolFormatDone`/`toolFormatFailed`
- "Backed up" badge — a green round shield-check (`icon_backup.svg`) for TigerTag+ whose signature is backed up. Shown as a round badge stacked with the twin "linked" badge (link on top, shield below). Grid cards + table thumbnails use one shared `_cardTlBadges()` cluster pinned top-left of the illustration (compact variant on the tiny table thumbnails); the detail panel keeps its own top-right cluster. The table thumbnail shows the twin only — the backup shield already appears inline next to "TigerTag+" in the Type column, so it isn't repeated there. In the unranked-spools sidebar (rack view), the backup shield is stacked directly UNDER the twin "linked" badge in the thumbnail cluster (via `thumbHTML`/`_groupMemberCardInnerHTML` `backup`/`stackBackup` options) instead of inline below the tier badge. New `rfidBackup` on the row + in the row/group signatures so it updates live — `assets/svg/icons/icon_backup.svg`, `renderer/css/70-detail-misc.css`, `renderer/inventory.js` (`tierBadgeHTML`, `_cardTrBadges`, `thumbHTML`, `_groupMemberCardInnerHTML`, `unrackedRowHTML`, `normalizeRow`, `_rowSignature`), i18n `backupBadge`
- Restore TigerTag+ availability is now keyed to the chip's UID, not the current spool type: the `rfidBackup` inventory flag means "a backup EXISTS in rfidList for this UID" (kept true even after a TigerTag+ is erased and rewritten as a plain TigerTag), so the Restore action reappears on that UID once the chip is on a reader — `renderer/inventory.js` (`recordRfidChipScan`, restore-tool visibility)
- Restore TigerTag+ — a toolbox action that restores a chip to its original state by rewriting the raw backup payload (pages 0x04-0x27) straight from `rfidList/{UID}.backup`. No TigerTag SDK (we hold the exact bytes) — new main IPC `rfid:repair` writes raw pages, surgical (only pages that differ), with read-back verify. Hard-guarded to the chip whose UID matches the backup (never clones onto another chip); with two readers/two chips present, the renderer routes to the reader actually holding this spool's chip and the main process double-checks the UID. Toolbox row shown only for a TigerTag+ that has a backup AND whose own chip is on a reader; hold-to-confirm, inline result. `main.js` (`rfid:repair`, `_pagesFromBytes`), `preload.js` (`repairRfidTag`), `renderer/inventory.js` (toolbox tool `btnRepairPlus` + wiring)

## Fixed
- Rack slot hover tooltip: the remaining-filament bar now follows the app-wide colour code (red < 20 %, orange < 50 %, green ≥ 50 %) via `fillBarColor(pct)` instead of always showing the accent colour — `renderer/inventory.js` (`buildRackTooltipHTML`)
- "Remove from inventory" tooltip was stale — it still promised the spool would land in "Settings → Debug → Deleted, restorable from there", a soft-delete scheme dropped in v1.7.4 (the action is a permanent hard delete). Rewritten (now shown in its ⓘ bubble) to state plainly that the material is removed for good and that the physical chip keeps its data (erase/recycle it to reuse) — i18n `spoolMarkDeletedTip`
- Twin "linked" + "backed up" icons now show on a freshly-scanned TigerTag+ without closing/reopening the card. Two causes: (1) `hasTwinPair` was only set on the half of a twin pair that survives dedup, so the just-scanned/opened half (~50% of the time the other one) had no link badge — now marked symmetrically on both halves right after the rows are built (`_markTwinPairs`, applied in the live, cached and friend-view inventory paths); (2) the detail-panel refresh compared a structural signature that omitted `rfidBackup`, so when the backup landed a snapshot later the panel took the weight-only patch path and never rebuilt the shield — `rfidBackup` added to `_detailStructuralSig`. `renderer/inventory.js`
- Chip-dependent toolbox actions ("Restore TigerTag+", cloud "Graver RFID") now appear/disappear live while the detail panel is open, instead of only being evaluated when the panel was first built. Placing or removing a chip on a reader refreshes the open panel (scroll position preserved, guarded by a presence signature so unrelated card events don't cancel an in-progress hold) — `renderer/inventory.js` (`openDetail` `preserveScroll` opt, `_panelCardSig`/`_maybeRefreshPanelForCard`, `onRfidCardPresent`)
