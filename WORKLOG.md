# Worklog — v2.8.0 (in progress)

## Added
- **Social-profile links on the account.** The Edit-account modal gains a **Social links** editor (paste any profile URL, add/remove rows, auto-saved). Links are stored as an ordered list `socials` on the private `users/{uid}` doc **and mirrored to public `userProfiles/{uid}.socials`** (no rules change — both are owner-write). The **icon is inferred from the URL host** (X, Instagram, YouTube, TikTok, Facebook, LinkedIn, Twitch, Discord, GitHub, WhatsApp; **globe** fallback for any other site) — flexible, no fixed platform enum. A row of brand-coloured icon links shows on **your own banner** and a **visited friend's banner**, and the owner's links ride along in the **public wishlist snapshot** (`publicLists/{token}.ownerSocials`, whitelisted + deployed) for the web page. New brand SVGs (Instagram/YouTube/TikTok/GitHub/Discord/Twitch/globe) + `.icon-*` classes; `_socialMeta` / `_cleanSocials` / `_socialsRowHTML` helpers + the modal editor — `renderer/inventory.html`, `renderer/inventory.js`, `renderer/css/60-modals.css`, `renderer/css/70-detail-misc.css`, `firestore.rules`. New `editSocialsLabel`/`editSocialsPlaceholder` i18n. Propagated to Integration + TigerHub BRIEF.
- **Friend count on a profile ("X friends").** Each account now publishes its friends count to its public `userProfiles/{uid}.friendsCount` (owner-written when the friends list changes — no rules change, userProfiles is already owner-write / auth-read). Visiting a friend shows the count in their banner, and the **Add-friend** preview shows it as **social proof** when you look someone up by code. Friendship is bidirectional, so this equals the number of people who have that account as a friend. **For a PUBLIC account the count reads "followers"** (`followerCount`); for a private/normal one it reads "friends" (`friendCount`). **Own account:** the banner now carries the same badges too — **Public** / **Private** (so you see your own status) + a **Share** badge to copy your own invite link — plus your own follower/friend count. (Own count was already in the Friends panel + `telemetry/studio.friendsCount`.) — `renderer/inventory.js` (`_syncFriendsCountToProfile`, `switchToFriendView`, `renderFriendBanner`), `renderer/css/60-modals.css`. New `friendCount` plural i18n (9 locales).
- **Wishlist count pill on the "Lists" view button** — a small brand-orange badge in the corner of the Lists icon shows how many lists you have (the friend's shared-list count in friend-view), live-updated from the lists subscription. Mirrors the "To order" cart badge — `renderer/inventory.html`, `renderer/inventory.js` (`_updateListsBadge`), `renderer/css/70-detail-misc.css`.
- **Friend-view banner — relationship badge + shareable public link.** The friend banner shows ONE clean pill badge next to the name: green **"Public"** (👁) if the inventory is public, else **"Friend"** (👤). The old **"Read-only"** badge is gone. For public accounts, a clickable **"Share"** badge sits below and copies that person's invite link (`cdn.tigertag.io/friend/<code>`) to pass on to your other friends (flashes "Copied!"). Removed the now-unused `friendViewReadOnly` i18n key; new `friendViewFriend`. The visited friend's `publicKey` is read from `userProfiles` on entering the view — `renderer/inventory.js` (`switchToFriendView`, `renderFriendBanner`, banner click handler), `renderer/css/60-modals.css`. New `friendViewPublic` / `friendShareInvite` i18n (9 locales; reuses `listShareCopied` for the copied feedback).

## Fixed
- **A public friend showed as "Friend" on a cold-start quick-click.** Opening the app and immediately clicking a public account rendered the relationship badge as "Friend" (and landed on the wrong view) and stayed wrong until a manual back-and-forth. Cause: `switchToFriendView` seeded `isPublic` from `state.friends[].isPublic`, which is populated asynchronously at startup (still empty on a fast click), and the follow-up `userProfiles` read — the source of truth — updated `publicKey`/`friendsCount`/`socials` but **not** `isPublic`. Now that read reconciles `isPublic` authoritatively: fixes the banner, heals the cached friend record (next click is instant), and corrects the landing view mode while still loading — `renderer/inventory.js` (`switchToFriendView`).
- **Public accounts' follower count was never updated (offline owner).** The friend/follower count was only published by the OWNER's client when their friends list changed — but a public account gets auto-accepted friends **server-side while offline**, so its `userProfiles.friendsCount` stayed stale/absent (nothing showed on their profile). Now the **`autoAcceptFriendRequestForPublic` Cloud Function** recounts BOTH sides' friends (via the `count()` aggregation — idempotent) and writes `userProfiles.friendsCount` for each after every auto-accept. Studio also **force-publishes the count when you flip your account to public** (first-time compute) + repaints your banner — backend `functions/index.js` (deployed), `renderer/inventory.js` (public toggle).
- **Couldn't reorder friends while viewing a friend.** A too-broad `state.friendView` guard blocked the friends drag-reorder (sidebar chips + Friends panel) in friend-view. Reordering YOUR friends writes to YOUR own account, so it's now allowed in friend-view too — the guard is removed from `_applyFriendReorder` and both wirers (the cart / printers / racks reorder still stay blocked, as those are the friend's read-only data) — `renderer/inventory.js`.
