# Worklog — v2.9.2 (in progress)

## Added
- **Debug "copy view ref" tool (admin debug mode).** ⚠️ _Internal / admin-only — do NOT include in the What's New (`data/whatsnew.json`) nor the release note (`data/release-notes/`); CHANGELOG only._ With debug mode on, **⌥(Alt)-clicking** any card / panel / view copies a paste-ready descriptor to the clipboard — the current view mode (+ friend-view), the surface it belongs to (human name · DOM id · render function · file, via a `_DBG_SURFACES` registry that tags every side-card panel — spool detail / product card / reorder / grouped-spools / printer / settings / notifications / friends / Firebase-explorer and the main views/modals), the data ids in scope (product hash, list id, selected spool/list), **one `openCard:` line per open side-card** (each stacked panel listed independently, even if the click landed elsewhere) and the clicked element — so the exact surface can be pointed at precisely. A small hint pill (`#dbgRefHint`) appears while debug is on (click it to copy the current view's ref), PLUS a **per-side-card copy button** (`.dbg-card-btn` — a bright pill at the bottom-right of each panel **showing that card's short name**, e.g. "🐛 Product card", injected/kept-alive by a MutationObserver) that copies THAT card's ref on click — because the bottom-right pill gets covered by an open side-card, and the on-pill name makes each card instantly identifiable. The copy is confirmed with the shared green "Copied!" flash (`_flashCopied`), same as SKU/EAN copy. Admin-only dev tool, English-only (refs are technical). `_dbgBuildRef` / `_dbgRefHintSync` in `applyDebugMode` — `renderer/inventory.js`, `renderer/css/70-detail-misc.css`.
- **Info button in the product side-card toolbox → opens the in-app reorder / To-order side-card.** The product business card (`openProductCard`, shown when clicking a list/favourite item) gains an info button next to the add-to-list / cart / favourite actions that opens the product's reorder/To-order side-card (`openReorderPanel` — editable price / buy-link / min-stock) **alongside the product card, without closing it** (both stack, arranged by `_syncPanels`). Owner-only (hidden in friend-view). The external TigerTag catalogue link (`tigertag.io/pages/product-infos/<id>`) stays as a clickable "Product ID" row in the card's details — `renderer/inventory.js` (`_renderProductCard` + `#productCardBody` click wiring → `closeProductCard` + `openReorderPanel`), `renderer/css/70-detail-misc.css` (`.flag-toggle--info`). New `productInfoPage` i18n (9 locales).
- **Drag-and-drop organisation of the lists sidebar.** Owners can now reorder their lists by dragging (persisted `sortRank`), AND drag a list from one visibility group to another (Private / Friends / Public) to **change its type** — the drop reconciles the public snapshot automatically (publish on → public, tear down on → private, via the existing lists-snapshot sync). All three groups always render as drop zones (each with a grab-grip on its rows; an empty group shows a dashed "Drop a list here" hint that lights up while dragging). Three-zone "make-room" DnD mirroring the cart, with an optimistic re-render + batched Firestore write (`_applyListDrop` / `_persistListOrder` / `_listZoneIdsFromDom`, delegated on `#invListsView`) — `renderer/inventory.js`, `renderer/css/70-detail-misc.css`. New `listTypeDropHint` i18n (9 locales). Friend-view stays read-only (no drag).

## Changed
- **"To order" view: clicking a material opens the product side-card.** In the To-order cart, a click on the material itself (thumbnail / name / row body) now opens the product business card (`openProductCard`) — the same entry point as a wishlist row, whereas before a line-body click did nothing. The ⓘ button still opens the reorder side-card, and the quantity selector, drag grip, buy / add-price / copy-SKU and set-aside controls keep their own behaviour — `renderer/inventory.js` (`#invProductsView` click handler, `.pv-order-line` branch).
- **A single gradient "type badge" now identifies list visibility everywhere.** One pill badge (same family as the TigerTag+ / TigerCloud badges — gradient fill, white icon + label, one colour per type: purple Private / blue Friends / green Public) is used consistently across the three surfaces where a list's type appears: the **"Your lists" sidebar group headers**, the open-list **Details card**, and (as coloured cards) the create/edit **type selector**. The Private glyph is the eye-off used everywhere else (the lock icon read as "secured/encrypted", misleading for a personal list), Friends = person, Public = globe. `.lv-side-group--{private,friends,public}` (badge), `.lv-type-badge--{…}` — `renderer/inventory.js` (`_listVisMeta` private icon → `icon-eye-off`), `renderer/css/70-detail-misc.css`.
- **Open-list view: a "Details" recap card in the right rail replaces the scattered list header meta.** A dedicated `.lv-info-card` above the Payment summary now consolidates the list's info: a header row ("Details" + an icon-only **Manage list** button), a compact two-up strip with the **type** shown as a gradient pill **badge** and the **item count** (icon + "3 Items"), then the **event** and **message** rows (icon + label, value below). Each field uses a small coloured accent icon — **no icon "boxes"** (rounded-square backgrounds read as heavy chrome), just the bare glyph (event = amber gift, message = blue envelope, type = group colour, count = muted). This frees the view header, which is now just the list name: the type badge, the item-count pill and the edit action all moved into the card (previously the occasion/message rendered bare, label-less, under the title). Removed the now-unused `.lv-actions` / `.lv-act` / `.lv-count` / `.lv-occasion` / `.lv-message` — `renderer/inventory.js` (`renderListsView`, `infoCardHTML` + slimmed `lv-head`), `renderer/css/70-detail-misc.css` (new `.lv-info-head` / `.lv-info-edit` / `.lv-info-stats` / `.lv-info-ic--{private,friends,public,count}` / `.lv-info-val--{private,friends,public}`).
- **Disambiguated the two meanings of "items" on the open-list screen.** The Details card counts distinct products (e.g. 3) while the Payment summary sums quantities (e.g. 17) — the shared word "items"/"articles" made "3" and "17" look contradictory. The Payment quantity counter (`pvItems`, used by both the list and the To-order summaries) is renamed to **"units"** (unités / Einheiten / unidades / unità / szt. …), so the Details "Items: 3" and the Payment "17 units" no longer collide on the same word — `renderer/inventory.js`, 9 locales.
- **List (row layout): per-item actions moved to the right of the row, shortening each line.** The quantity selector + buy-link + remove button used to stack as a fourth row under the thumbnail/title/price block, making each line ~140px tall; they now sit on the right of the row, vertically centered on a single line, so the height is driven by the thumbnail (~113px, ‑20%). No element resized — only relocated (`.lv-row-actions` lifted out of `.lv-row-body`; `.lv-row` gets `align-items:center`) — `renderer/inventory.js` (`_listRowHTML`), `renderer/css/70-detail-misc.css`.
- **List grid card: the buy-link button moved below the quantity** (its own full-width row) instead of sharing the row — the card is too narrow to fit both. A long shop domain now **ellipsizes** inside the button instead of spilling out (the label text is wrapped in `.lv-buy-txt` with `min-width:0` + `text-overflow:ellipsis`; the cart icon stays fixed). `.lv-card-buybar` stacks vertically — `renderer/inventory.js`, `renderer/css/70-detail-misc.css`.
- **Create/edit-list modal type selector: matching icons + a colour per type.** The Private card's padlock (`icon-lock`) is replaced by the eye-off glyph used everywhere else for a private list, and each of the three type cards now carries its own accent colour (purple Private / blue Friends / green Public) on its icon and its selected state — instead of all three sharing the generic orange `--primary` — so the type language is consistent with the sidebar and the Details badge — `renderer/inventory.html` (`.clm-vis-opt[data-vis="private"]` icon), `renderer/css/60-modals.css` (per-type `--vc`).
- **Create/edit-list modal reworked around the list type.** The visibility choice moved from a dropdown at the very bottom to a prominent **3-card segmented selector** (Private / Friends / Public — icon + title + one-line description) placed right under the name, since it's the structuring decision. The message field is **always shown but reframes by type** — "Message for your friends" on a shared (Friends/Public) list, or "A note to future you" on a Private one (`_clmSetVis` swaps the label + placeholder). New-list default is now **Private** (was Friends), matching the privacy-first copy. The old `<select id="clmVisibility">` became a hidden input driven by the cards (`_clmSetVis`); dropped the redundant help paragraph — `renderer/inventory.html`, `renderer/inventory.js` (`openCreateListModal`, `_clmSetVis`, `_submitCreateList`), `renderer/css/60-modals.css` (new `.clm-vis*`).
- **The "Your lists" sidebar is grouped by visibility** — lists are bucketed under Private / Friends / Public section headers (in that order, only non-empty groups shown). The **Public** visibility icon changed from an eye to a **globe** (`icon-globe`) — public = shareable on the internet — across the list group header, the title badge and the public-share card — `renderer/inventory.js` (`renderListsView`), `renderer/css/70-detail-misc.css` (new `.lv-side-group`; `.lv-side-n` restyled as a pill; removed the now-unused `.lv-side-vis`).

## Fixed
- **Cam view showed the same Creality camera for every connected Creality.** The Creality WebRTC widget was a global singleton — one peer connection, one stream, and a consumer set populated by `document.querySelectorAll(".cre-cam-video")` regardless of which printer each `<video>` belonged to. The cam wall also only started the stream for `crePrinters[0]`. So every Creality card in the Cam view received the first (or last-started) printer's feed; the sidecard looked correct only because it shows a single video at a time. Reworked the widget to keep **one WebRTC session per printer IP** (`_sessions` map), route each `<video>` to its own session via a new `data-cre-ip` tag, start a session for **every** online Creality, and stop sessions per-IP (so disconnecting one Creality no longer tears down the others). The reconnect path stamps `data-cre-ip` on the sidecard video (the panel isn't rebuilt on reconnect) — `renderer/printers/creality/widget_camera.js` (rewrite), `renderer/inventory.js` (cam-wall loop, per-IP `stopCreCam`, `creCamStop`, sidecard `data-cre-ip`), `renderer/printers/creality/index.js` (reconnect stamp + per-IP stop), `renderer/printers/context.js` (`creCamStop` signature).
- **Lists view couldn't scroll with many items — each column now scrolls on its own.** The open-list view grew with its content but `#card-inv` clips overflow, so a long list (or a tall right rail) was simply cut off with no way to scroll. The view now fills the remaining height and splits into three independently-scrolling columns — the lists sidebar, the item list (its header stays pinned while the rows scroll), and the right rail (Details / Payment / public-link card) — mirroring the products view's `flex:1; min-height:0; overflow-y:auto` pattern. The rail's old `position:sticky` is dropped in favour of its own scroll, and the scrollbars are hidden (`scrollbar-width:none`) so scrolling stays but the chrome doesn't, matching the products view — `renderer/css/70-detail-misc.css` (`.inv-lists-view` / `.lv-amz` / `.lv-side` / `.lv-main` / `.lv-main-col` / `.lv-rows` / `.lv-rail` / `.lv-head`).
- **Two spools could end up assigned to the same rack slot — now auto-healed + prevented.** Rack-slot uniqueness was enforced client-side only (Firestore has no such constraint), so a cross-device race or Auto-organize running against an incomplete/cache snapshot could drop a second spool onto an already-occupied slot (one physical, one TigerCloud in the reported case). Added `healDuplicateSlots(uid)` — on each authoritative, settled SERVER inventory snapshot it detects any slot holding >1 spool-unit (a linked twin PAIR counts as one) and evicts the extras to "not stored" (`rack: null`), deterministically (keeps the smallest spoolId so every device agrees) and idempotently. GUARD: **Auto-store now only runs on a SERVER snapshot** (never on cache/partial, where a not-yet-loaded occupant looked absent) — the root cause of new duplicates. The evicted spool re-files itself into a genuinely-empty slot on the next pass — `renderer/inventory.js` (`healDuplicateSlots` + `subscribeInventory` snapshot handler).
- **Lists view — grid cards now open the product side-card on click, like the rows do.** The Lists-view click handler only matched `.lv-row[data-hash]`; grid cards (`.lv-card[data-hash]`) fell through and did nothing. Extended the selector to match both (the interactive-control guards — buy / qty / remove / add-price — already cover both layouts) — `renderer/inventory.js`.
