# Worklog — v2.11.0

## Added
- **"To order" line: a "Stop tracking" action to remove an item from the reorder list.** Until now, once a product was in the reorder list there was no way to take it out — "Buy later" only moves it to the saved shelf, it stays tracked. Each line (cart + saved shelf) now has a **Stop tracking** press-and-hold button (1 s, red fill sweep) that resets the product's minimum stock to 0, which clears the whole reorder state (min / savedForLater / orderQty) and drops the line from the To-order view entirely. Hold-to-confirm (not a plain click) since it discards the min setting — `renderer/inventory.js` (`_orderLineHTML` `stopLink` with `.hold-progress`, wired via `setupHoldToConfirm(btn, 1000, → _writeProductField(hash, { minStockSpools: 0 }))` after the order-tab render; a plain click is swallowed), `renderer/css/70-detail-misc.css` (`.pv-move-link--stop`). New `reorderStopTracking` i18n (9 locales).
- **Loading spinner on the Favorites / To-order, Lists and Storage views (no more empty-state flash).** These views read their data from Firestore subscriptions, so before the first snapshot arrived they briefly showed "nothing here" / "no lists yet" / "no racks yet" and then popped the data in — most visibly when opening a friend. Added `state.productsLoading` / `state.listsLoading` / `state.racksLoading` (default **true**, and set true again whenever the matching subscription (re)starts — until its first snapshot, own or friend; defaulting true means even the very first render, which can precede the subscription wiring, shows the spinner instead of a split-second empty state), and the render functions now show the shared spinner (`.inv-loading`, generic `loadingGeneric` copy) while loading and only fall through to the racks / empty illustration once the data has actually arrived. For Storage the gate is `state.racksLoading` in your own view and `state.invLoading` in friend-view (the friend's racks are fetched as part of the one-shot friend read). A denied/errored read drops the spinner (shows the empty state) instead of spinning forever; a friend visit doesn't flash a spinner on the way back (own data stayed loaded); the own subscription can't clear the friend's flag while in friend-view — `renderer/inventory.js` (`subscribeProducts/Lists/Racks` + friend variants, `renderProductsView`, `renderListsView`, `renderRackView`, `switchBackToOwnView`). New `loadingGeneric` i18n (9 locales).
- **"To order" cart: a personal note per purchase source.** Each source card gains an editable free-text note under its header (e.g. "also grab 3 nozzles + a bed cleaner") — a subtle amber strip when set, a quiet "Add a note" prompt when empty. The note strip has square corners (the `<button>` "Add a note" prompt would otherwise inherit the global button `border-radius`, leaving rounded notches at the card edges). Click to edit inline (textarea; Enter saves, Shift+Enter for a newline, Escape cancels); an empty note is cleared. Stored locally keyed by host (`tigertag.cartSrcNotes`), so it follows a reordered card. Owner-only (read-only in friend-view) — `renderer/inventory.js` (`_getCartSrcNotes`/`_setCartSrcNote`/`_startCartNoteEdit`, note in `_renderOrderTab`, `#invProductsView` click handler), `renderer/css/70-detail-misc.css` (`.pv-cart-note*`). New `cartAddNote` / `cartNotePh` i18n (9 locales).
- **"To order" cart: reorder the purchase-source cards by drag-and-drop.** Each source card's header has a grip; dragging it reorders the source groups, and the order is remembered (localStorage `tigertag.cartSrcOrder`). The no-link card stays pinned last; new sources appear after the saved ones (alphabetical). The grabbed card's drag preview keeps rounded corners (the card drops `overflow:hidden` — which made the drag image square — and rounds its border + top/bottom inner sections instead). Reuses the shared make-room DnD helper, gated to the header grip (`.pv-cart-grip`) so it never collides with the existing within-card line reorder (`.pv-grip`) — `renderer/inventory.js` (`_getCartSrcOrder`/`_setCartSrcOrder`, `_cartKeys` sort, `_wireMakeRoomDnd` on `#invProductsView`), `renderer/css/70-detail-misc.css` (`.pv-cart-grip`, `.pv-cart-card--dragging`).
- **"To order" cart: each purchase source is now its own card with a subtotal footer.** The cart was one continuous list with sticky source headers; it now renders each purchase source (buy-link host) as a **separate bordered card** — header (source + count), its lines, then a **subtotal footer** at the bottom of the card (Σ unit price × qty over its priced items, in the account's HT/TTC mode; shown only when the group has ≥1 priced item). The footer states the tax mode with a localised **HT / TTC** tag (`reorderHT`/`reorderTTC` — "excl./incl. tax", "netto/brutto", …) after the amount, so it's clear whether the subtotal is tax-excl or tax-incl per the user's config. Separating each order visually, and complementing the global Payment card. Hovering a line keeps its bottom divider visible inside a card (the generic order-line hover turned the border transparent, which hid the last line's separator from the subtotal footer). The drag-to-reorder is unchanged (still group-constrained; lines stay reorderable within their card) — `renderer/inventory.js` (`_renderOrderTab`: `.pv-cart-card` wrapper + `.pv-cart-card-foot`, cart zone → `.pv-cart-cards`), `renderer/css/70-detail-misc.css` (`.pv-cart-cards` / `.pv-cart-card` / `.pv-cart-card-foot`; `.pv-cart-group` no longer sticky).

## Changed
- **Grouped-spools side-card: the deck actions became one icon-only row with favourite + to-order toggles.** The panel now exposes the same flag toggles as the product card so you can favourite (★) and add to the reorder cart (❤ "to order") straight from the deck — previously it only had Product-info + Buy + a text "Add to a list". The row is now icon-only: **Product-info on the left**, then the three toggles grouped on the right — **add to cart (❤), favourite (★), add to a list (＋)** — each a `.flag-toggle` reflecting/writing the product's state via `_toggleProductFlag`. The three toggles also render in **friend-view** (where they import the friend's product into your account — provenance + price/link carried by `_toggleProductFlag`), but **without** the Product-info button (owner-only, and the below-min-stock alert + read-only note stay owner-only too); with no info button the toggles stay right-aligned (`justify-content: flex-end`). The old text "Add to a list" (own row) and the standalone Buy button are gone (the shop link stays reachable from the reorder side-card the info button opens) — `renderer/inventory.js` (`_renderGroupPanelContents`), `renderer/css/70-detail-misc.css` (`.gp-actions--flags`).
- **"Add to a list" now works in friend-view — it adds a friend's material to one of YOUR lists.** The whole lists flow was owner-gated, so the ＋ button on the friend-view grouped-spools side-card did nothing. The "Add to a list" popup now always targets **your own** lists (new `_ownLists`/`_ownListsArray`/`_ownListHas`, used for the popup rows + tick state instead of `_listsSource()` which flips to the friend's lists), and picking a list **imports** the friend's product into your account first — carrying their shared price / buy link / SKU-EAN + stamping provenance, same as favouriting — before adding its key to your list, so the item shows up with a picture, price and buy button back in your own view. Creating a new list on the spot from a friend's item works too. Removed the `state.friendView` early-returns on `_addToList` / `_removeFromList` / `_createList` / `openCreateListModal` / `_submitCreateList` / `_openAddToListMenu` (all write to your own account via `getActiveId`; the Lists-view edit/new controls remain unrendered in friend-view, so no friend list can be mutated). The ＋ button now also shows in friend-view on the **grouped-spools side-card** and the **spool detail panel** (next to ❤/★), not just the product card — `renderer/inventory.js` (`_flagTogglesHTML`, `_renderGroupPanelContents`).
- **Loading animation restyled to a bigger comet-trail loader.** The shared `.inv-loading-spin` (Inventory / Favorites / To-order / Lists loading states) is now a rotating comet — five `box-shadow` dots forming a fading tail that winds up and slings around (`load6` + a dedicated `round` 0°→360° keyframe; adapted from Luke Haas' css-loaders, MIT), brand-orange (the reference's white would be invisible on the light background), drawn from a single element so no markup changed. Under it, a **carousel of playful phrases** ("Waking up the spools…", "Warming up the bed…", "Time for a coffee?", "Almost there, promise…") that slide in from the right and out to the left, one at a time, with both edges fading through a gradient mask. All loading states share one `_loadingHTML()` helper, shown via `_showLoading(host)` which only builds the loader if it isn't already present — so a re-render while still loading (cache→server snapshot, badge refresh…) no longer rebuilds the DOM and restarts the spinner + phrase carousel half a second in. The phrase order is shuffled (Fisher-Yates) each time the loader first appears (Inventory / Favorites / To-order / Lists / Printers). The cam-wall keeps its own small border spinner — `renderer/inventory.js` (`_loadingHTML` + 5 render sites), `renderer/css/70-detail-misc.css` (`.inv-loading-spin` + `load6`, `.ld-phrases` + `ld-phrase`), `renderer/css/40-printers.css`. New `loadingFun1`–`loadingFun4` i18n (9 locales).
- **"To order" view: the Payment card top-aligns with the first source card.** It used to line up with the "Panier" section title above the cart, sitting higher than the first card; a `margin-top` matching the title's height drops it to the first card's top (reset in the stacked mobile layout) — `renderer/css/70-detail-misc.css` (`.pv-order-side`).
- **Open-list Details card: the Event (gift) and Message (mail) icons are larger** (icon-16 instead of icon-12) so they read more clearly next to their labels — `renderer/inventory.js` (`infoCardHTML`).
- **Product card: the buy link sits directly under the price, styled like the reorder card's shop button.** The buy link was a detached grey button below the price section; it now lives inside the same price section (directly under the price line) and adopts the reorder / To-order card's active-shop-button look — Shopify green with a white cart icon + shop name — for a consistent buy affordance across the two side-cards — `renderer/inventory.js` (`_renderProductCard`), `renderer/css/70-detail-misc.css` (`.pc-buy`, `.pc-rows`).

## Fixed
- **Sidebar "Mobile Apps" QR flashed a broken-image icon on cold start.** The sidebar QR `<img>` ships with no `src` (the code is generated locally and set on it after the renderer module loads), so for a beat on launch the browser showed its broken-image placeholder + the "QR" alt text. The image is now hidden until it has a `src` (`.sb-qr-img:not([src])`), its fixed box keeping the layout stable, so the QR just appears once generated — `renderer/css/10-settings.css`.
- **Friend with an empty inventory: the decorative rack lingered after leaving the Storage view.** Visiting a friend who has nothing in stock and switching from the Storage (rack) view to the Inventory table/grid left the empty-rack illustration — plus its stats pill and the "unranked spools" side-panel (all rendered inside `#invRackView`) — visible behind the "nothing here" empty state. The friend-view empty-inventory branch of `renderInventory` hid the table/grid containers but not `#invRackView` (nor the printer/products/lists containers); it now hides them all, mirroring the own-view empty branch — `renderer/inventory.js` (`renderInventory`).
- **Lists count badge lingered on the wrong context (own ↔ friend).** The red count pill on the "Lists" view button is context-aware (`_updateListsBadge` reads your lists, or the friend's in friend-view), but it was only refreshed by the lists snapshots — which don't fire on a view switch, and never fire at all when a friend's lists read is denied (a non-shared friend account) or arrives empty. So it kept the previous count: your own count when opening such a friend (badge stuck on e.g. "4"), and the friend's count after switching back. Fixed at both transitions — `switchBackToOwnView` recomputes it from your own lists after clearing friend-view, and `subscribeFriendLists` recomputes it up front from the (reset) friend context before the async snapshot — `renderer/inventory.js` (`switchBackToOwnView`, `subscribeFriendLists`).
- **Right-click menu (Cut / Copy / Paste / Select All) now follows the in-app language, not the OS locale.** The native context menu was built in the main process from Electron `role`s, whose labels come from the OS locale — so a user with the app in one language but the OS in another saw the wrong language. The renderer now pushes the translated labels (`ctxCut` / `ctxCopy` / `ctxPaste` / `ctxSelectAll`) to the main process on startup and on every language change (via a new `app:ctx-menu-labels` IPC + `electronAPI.setContextMenuLabels`), and the menu overrides each role's label with them (falls back to the role default until the first push) — `main.js`, `preload.js`, `renderer/inventory.js` (`applyTranslations`). New i18n keys (9 locales).
