# Worklog — v1.8.23 (in progress)

## Added
- Centralised avatar rendering pipeline — one source of truth for every coloured-circle avatar in the app (sidebar, top "OM" header chip, edit-account modal, account dropdown, profiles modal, friends list in the sidebar, friends panel, friend banner). Two public entry points: `paintAvatar(el, source)` for elements updated in place and `avatarMarkup(source, className, extraClass)` for innerHTML-built lists. Both delegate to `_buildAvatarParts` which computes gradient + initials + photo overlay atomically. Initials are now derived STRICTLY from `displayName` — never the email prefix, which was producing wrong "B from benoit@…" letters during the boot window before Firestore resolved. DOM structure is now uniform across every avatar variant (`.av-initials` text span + optional `.sb-avatar-photo` `<img>` overlay), and `paintAvatar` updates the initials without nuking the photo overlay or any other siblings. Friend objects' single-hex `color` field is promoted to a `customColor`-style subject so friend chips render with the same 135° gradient as own avatars instead of a flat-colour fallback. — `renderer/inventory.js`, `renderer/inventory.html`, `renderer/css/00-base.css`, `renderer/css/20-friends.css`, `renderer/css/60-modals.css`.
- Discord-style instant-paint hydration — `state.photoURL`, `state.friends`, and `acc.displayName` are now hydrated synchronously from `localStorage` in `setConnected` BEFORE the first paint. The result: on app open, the sidebar avatar shows the user's actual photo or initials immediately (no flicker from "empty → wrong letter → right letter → photo"), and the friends list chips + dropdown show every friend with their photo straight away. `loadFriendsList` mirrors the minimal friend objects (uid + displayName + color + photoURL) to `tigertag.friends.<uid>` localStorage, hydrated back by a new `_hydrateFriendsCache` helper at connect time. — `renderer/inventory.js`.

## Changed
- `syncUserDoc`'s `userProfiles.photoURL` fetch is now idempotent — only updates `state.photoURL` (and re-renders) when the URL actually changed since hydration. The fetch failure path no longer nulls `state.photoURL` either — transient Firestore blips keep the cached avatar visible instead of flashing the user back to their initials.
- Friends-list `loadFriendsList` now also refreshes each friend's `photoURL` from `userProfiles` on every cycle (stored in memory only — the Firebase Storage URL's rotating token would force a fan-out on every avatar change if denormalised into the friend doc).
- File picker for avatar upload (`_pickAvatarFile`) now uses the modern `cancel` event on `<input type="file">` (HTML-spec, Chromium 113+, fires reliably on Win10/Win11/macOS/Linux) instead of the legacy `focus`-listener-with-grace-timer pattern, which was racing the `change` event on Windows 10's I/O scheduler and silently resolving null — the cropper never opened. The focus listener is kept as a belt-and-braces fallback with a longer 800 ms grace window. Reported by a Win10 user as a hard bug: image picked, cropper never opens, no error. — `renderer/inventory.js`.

## Fixed
- "OM+" badge bleeding through the sidebar avatar — the `.sb-avatar-plus` "+" affordance (only meant for the signed-out state) was leaking next to the user's initials in the signed-in state. Root cause: `.sb-avatar-plus { display: none }` in `00-base.css` and `.icon { display: inline-block }` in `70-detail-misc.css` had the same CSS specificity (0,1,0), and `00-base` is loaded first → `.icon` won the cascade. Specificity bumped to `(0,2,0)` via `.sb-avatar .sb-avatar-plus { display: none }`. — `renderer/css/00-base.css`.
- Google profile-picture URL clobbering the user's custom avatar — `onAuthStateChanged` was writing Firebase Auth's `user.photoURL` (a `lh3.googleusercontent.com` URL pointing at Google's auto-generated "letter-on-coloured-circle" placeholder for accounts without a real Google photo) into `acc.photoURL` on every sign-in, overwriting the custom Firebase Storage avatar URL the user had uploaded. The visible symptom was a Google-rendered "B-on-violet-circle" placeholder appearing in the sidebar avatar after a sign-out / sign-in cycle. Now `user.photoURL` is no longer read at all; `userProfiles.photoURL` (synced from Firebase Storage) is the sole source of truth for the custom-avatar URL. A one-time legacy migration in `setConnected` also clears any pre-existing Google-CDN URL from `acc.photoURL` for users who hit the bug before this fix. — `renderer/inventory.js`.
