# Worklog — v2.13.1 (in progress)

## Added

- **Debug-only reader readout in the TigerPOD modal** (admin/debug mode) — in place of the removed ownership toggle, it shows what the app currently treats as an **active** reader (name + whether a chip is on it), plus the account's TigerPOD telemetry (lifetime max reader count, `hasPod`). Reads live state (`state.nfcReaders`, `state.nfcCardPresent`, `_telRfidMax`) and **refreshes in real time** — the reader plug/unplug and card present/removed handlers re-render it while the modal is open (guarded so it's a no-op when closed), and it also repaints when the telemetry read lands — `hydratePodSignal` is a fire-and-forget read at connect, so a modal opened before it returned showed the pre-hydration defaults (max 0, hasPod false) until now. Shows current vs max reader count side by side. Hidden entirely for non-debug users. Admin/dev surface — not user-facing — `renderer/inventory.js`, `renderer/inventory.html`, `renderer/css/60-modals.css`

- **Reader-selection panel brief** — `docs/READER-SELECTION-BRIEF.md`, the frozen spec for a per-reader active/inactive panel: the set of active readers is what the app considers usable (burn flow, card-present, "no reader" prompt all key off it). Smart defaults fold the YubiKey filter in (security keys default off with the reason shown, real readers default active), choices persist per reader, and it handles the two-identical-ACR122U case. Ships the immediate YubiKey fix now; the panel is the general control — `docs/READER-SELECTION-BRIEF.md`

- **`.ttag` export/import brief** — `docs/TTAG-EXPORT-BRIEF.md`, the frozen spec for backing up inventory materials to a `.ttag` file so they live outside the cloud. A `.ttag` is a verbatim copy of the Firestore doc(s), `records[]` of 1→N (one spool / a twin / the whole inventory, no branching); twins are atomic (exporting one always exports both) and re-paired from `twin_tag_uid` in the data; import validates then previews then accepts, with restore (same account, verbatim) vs import (other account, ids regenerated + twin refs remapped) modes. Frozen identifiers (`.ttag`, UTI `io.tigertag.ttag`, ProgID `TigerTag.ttag`, content marker), and a Phase 1 (export + file-picker import, testable in dev) / Phase 2 (OS association, needs a build) / Phase 3 (whole-inventory) split. The signed TigerTag+ backup (a separate `rfidList` collection, not the inventory doc) rides along in a required `rfidBackups` map so the backup is complete. A multi-material export asks grouped (one file, all records) vs individual (one file per material); a twin is 2 records but one material, so it stays one file and counts as individual — `docs/TTAG-EXPORT-BRIEF.md`

- **Length-limited fields now shake when they reject a keystroke** — a `maxlength` field used to swallow the over-the-limit character in silence; on desktop the equivalent of a phone's haptic "no" is a quick horizontal shake, so the rejection is seen. One delegated `beforeinput` listener covers **every** `[maxlength]` input and textarea in the app (13 today, from the 3-char friend-code boxes to the 500-char list message), present or added later, so nothing is wired per field. It fires only on a genuinely rejected insert — a paste or keystroke that would overflow, accounting for any selected text it would replace. Honours `prefers-reduced-motion`: a single danger-tinted border flash instead of movement. Shared `bumpField()` helper + `fieldBump` keyframe, reusable for any future hard-limit feedback — `renderer/inventory.js`, `renderer/css/00-base.css`

- **Partner packaging assets** — `assets-src/img/partners/`, the retail boxes from eSun, SUNLU and ROSA3D whose spools carry the TIGER TAG RFID badge, in two variants each (a 1024 cut-out with transparent background for compositing, a 2000 studio shot for standalone use). These are the only assets in the project that prove *adoption* rather than describing the product. Masters only — no shipped copy under `assets/` yet, since the use is not decided. The folder README flags the point that matters before any of them is published: they show other companies' trademarks, so factual use ("filament makers shipping TigerTag") is normally fine while implying partnership is not, and written permission is worth having on file — `assets-src/img/partners/`

## Changed

- **The stock-value tile shows its HT/TTC qualifier next to the price, not under the label** — `236.60 € excl. tax` instead of `236.60 €` with `STOCK VALUE · excl. tax` below. It qualifies the figure, so it belongs on the figure's line. The number is wrapped in its own span so the count-up tween (which sets `.value` textContent) no longer wipes the suffix; the qualifier has its own muted colour so the up/down tint never turns it green/red — `renderer/inventory.js`, `renderer/css/00-base.css`

- **New chipless spools are minted with a `TigerData_` id prefix** (was `CLOUD_`), so an exported `.ttag` never shows a confusing `CLOUD_` id — the file carries the tier name the user knows. The prefix is a *discriminator*, not a display value: `isCloud = spoolId.startsWith("CLOUD_")` decided the whole chipless path. Rather than migrate (a Firestore doc-id rename is delete+recreate across every account) the discriminator now accepts **both** prefixes via a shared `_isChiplessId` — `CLOUD_` stays recognised forever for the docs already out there, `TigerData_` is used going forward. Studio has a single discriminator point (`normalizeRow`) and a single mint point (`_adpCloudId`), both updated. The filter chain is unchanged: `TigerData_` → `isCloud` → `protocol:"TigerCloud"` → shown as "TigerData", so it lands under the existing search selector. **This prefix is read by four codebases** — the same both-prefixes generalisation was applied to the mobile app (`main_inventory.dart`), the Hub (`lib/data/inventory.ts`) and the stats Cloud Function (`functions/index.js`, where `cloudCount` would otherwise under-count), plus the `llms.txt` / backend-README docs. **Deployment order matters**: a `TigerData_` spool created in Studio syncs to Firestore and is read by the phone / Hub / Function immediately, so their both-prefix updates must be live before the Studio release that flips the minting reaches users, or an old client reads a new spool as a real chip and shows the wrong tier — `renderer/inventory.js` (+ mobile, Hub, backend)

- **Standing review brief** — `docs/REVIEW-BRIEF.md`, a read-only audit brief covering security, bugs, performance, first-try UI correctness, development velocity / token cost, and the brand-licence-certification foundations behind selling chips to manufacturers. - **Documented that URL fields are attacker-controlled, and propagated it to every client.** The two stored-XSS holes were not a renderer bug but a property of the data model: `buyUrl`, `attachments[].url`, the inventory `Link*` fields and `publicLists` items are written by a user into their own document and read by others — `publicLists` by anyone, unauthenticated. The Firestore rules cannot filter this (a `hasOnly()` whitelist controls *which* fields may be written, never what a string contains) and a client-side sanitiser is bypassed by writing to Firestore directly, so the check has to live in each renderer. Written up in `docs/firestore-schema.md`, in the backend README and in the third-party integration guide — including the part that caused the bug: HTML-escaping is not this check, it stops tag injection and leaves `javascript:` in an `href` intact. Audited the other clients while there: Tiger Hub was already safe on `publicLists`/`products` but not on `inventory` (three copies of `cleanUrl`, two behaviours — now aligned), and the mobile app now scheme-checks before `launchUrl`, which matters once manufacturers write catalogue entries — `docs/firestore-schema.md`

## Fixed

- **A YubiKey (or other Yubico security key) is no longer treated as an RFID reader** — plugged in over USB it exposes a PC/SC CCID interface, so `nfc-pcsc` enumerated it exactly like an ACR122U and the app tried to use it as a filament reader (reported by a real user). The reader-registration gate now skips devices whose PC/SC name matches `yubico`/`yubikey`. Deliberately narrow (Yubico only) to avoid excluding a real reader with an unusual name; the general user-controlled mechanism is the reader-selection panel (`docs/READER-SELECTION-BRIEF.md`) — `services/nfc-process.js`

- **The Add-Product panel's NFC Data preview is now filled on open** (debug mode). The panel builds the preview whenever a field changes, but on open every one of those refreshes ran *before* the section was un-hidden, so each hit the hidden-section early-return and did nothing — the field stayed empty until the first manual change. `openAddProductPanel` now runs one refresh right after un-hiding the section, so the default product (Generic PLA Basic, 1000 g) renders straight away — `renderer/inventory.js`

- **The step badges in the calibration modal's "How to measure" strip now sit at the top of every card** — the head was `align-items: center`, so a caption that wrapped to two lines (steps 2 and 3, "Keep the cardboard core" / "Put it all on the scale") dropped its orange number badge to the vertical middle while step 1's one-line caption kept its badge at the top. Switched to `align-items: flex-start` so the badge pins to the first line regardless of wrap, plus a −2px nudge so the 18px badge's centre sits level with the 14px caption line rather than 2px below it. Verified the three badges land at delta 0.00 from the first-line centre — `renderer/css/60-modals.css`

- **Printer connections no longer leak on account switch or sign-out** — `unsubscribePrinters` disconnected Elegoo, Bambu Lab and Anycubic but not Snapmaker, FlashForge or Creality, which were torn down only by the manual connect button. Signing in on one account, opening Printers so a FlashForge auto-connects, then switching accounts left its 2-second poll loop and the Snapmaker/Creality sockets running against the *previous* account's printer — and another set was added on every switch. All six brands are now swept, with the Creality camera stopped alongside its socket — `renderer/inventory.js`
- **`img:get` no longer fetches whatever it is handed** — it took a URL straight from a Firestore image field (including a friend's custom image URL, which they control), fetched it with no validation and no timeout, then wrote the body to the cache and served it back to the renderer: a server-side request forgery with readback, where an internal service answering on loopback became readable. It now refuses anything that is not a public http(s) destination, resolving the hostname first so a domain the attacker owns cannot point at 127.0.0.1, and rejecting loopback, RFC1918, link-local, CGNAT and IPv6 ULA. Redirects are followed by hand and re-validated at each hop rather than refused outright — refusing them would have silently broken the CDNs and short links real image URLs use. Added an 8-second deadline so a slow host cannot pin the handler open — `main.js`

- **Closed two stored-XSS holes reachable from another user's data** (found by the 2026-07-19 review). `esc()` escapes HTML but never validated the URL scheme, so `javascript:`, `file://` and `smb://` passed through every `href` untouched — and the values came from a friend's product links and attachments, which live in *their* Firestore doc and can be written directly, bypassing the client sanitiser. On click the payload ran in the renderer with the full preload bridge and `state.privateKey` in reach. New `safeHref()` composes the existing `_looksLikeUrl()` check with `esc()` and collapses anything that is not a plausible http(s) address to `#`. Applied at **all 8** `href` sites, not only the 3 the review traced — uniform means nobody has to work out which ones a friend can reach — `renderer/inventory.js`
- **Scheme allowlist and navigation lock in the main process** — `shell.openExternal` accepted any string on the `shell:open-external` IPC channel and in the window-open fallback, so a renderer foothold could hand the OS a `file://` path or an app-scheme. New `isSafeExternalUrl()` gates all three call sites and fails closed (anything not provably http(s) is refused, rather than blocking known-bad schemes by name). Added the `will-navigate` handler that did not exist: the renderer never navigates itself, so any attempt to leave the app's own origin is an injected link or a redirect — and the destination would inherit the preload bridge. Same-origin passes, external http(s) is handed to the browser, everything else is dropped. The OAuth call is left ungated: its URL is a hardcoded `accounts.google.com` constant with no external input — `main.js`
- **The low-stock check no longer re-scans the inventory once per product** — `_checkLowStockNotifs` runs on *every* render and filtered `state.rows` once per product, plus a second time for each product below its minimum, so its cost was products × rows every time. It now calls `_stockCountByKey()`, which already builds exactly that map in one pass and is what the rest of the order math uses — `renderer/inventory.js`

## Removed

- **The "I own a TigerPOD" toggle** in the TigerPOD modal, with its `tigerPodOwnLabel` / `tigerPodOwnSub` i18n keys and `.tigerpod-own-card` CSS. It was redundant: `hasPod` is a telemetry-only flag (`telemetry/studio.hasPod`) that **nothing in the app behaviourally depends on**, and it is auto-set on the first successful RFID read with readers — so a manual toggle changed nothing the next scan wouldn't re-set. The auto-set (and the whole 1-or-2-reader write path) is untouched — `renderer/inventory.html`, `renderer/inventory.js`, `renderer/css/60-modals.css`, i18n (2 keys × 9 locales)
