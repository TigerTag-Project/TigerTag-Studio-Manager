# Worklog — v2.13.2 (in progress)

## Added

- **`.ttag` export — a faithful offline backup of an inventory material** (Phase 1 of `docs/TTAG-EXPORT-BRIEF.md`). A new "Export .ttag" action in the spool detail toolbox writes a `.ttag` file (`{ format:"tigertag", kind:"ttag", version:1, exportedAt, exportedBy, records[], rfidBackups? }`): the inventory doc(s) carried **verbatim** (Firestore Timestamps normalised to epoch-ms so they serialise clean), plus — for a TigerTag+ chip — its signed `rfidList/{chipUid}.backup` doc, keyed by chip uid. **Twins are atomic**: exporting one side resolves and includes its `twin_tag_uid` partner (one material → 2 records, never a half-twin). **Invariant #2 enforced**: a legacy `CLOUD_` id is refused (button flashes "reconnect to finish") so it can never leave in a file — only the current `TigerData_` chipless prefix may. `exportedBy` (the owner uid) is stamped so import can auto-pick its mode. Own inventory only (hidden in friend view), no reader needed; renderer-side Blob download (no build), file named `brand-material-color.ttag`. Inline button feedback (label swap → "✓ Backed up"), the Duplicate-action pattern. **Plus a selection Export**: an "Export" button in the inventory toolbar (left of Import / "+ Material", same style) exports the **currently-selected materials** (multi-select mode) into one `tigertag-selection-<date>.ttag` (each selected spool + its twin partner, deduped, same CLOUD_ refusal). It is **disabled while nothing is selected** (`_syncTtagExportBtn`, driven off `state.selectedSpools` via `_updateBulkBar` / select-mode exit). Both toolbar buttons show only in the material inventory (table/grid) and hide on a friend view. — `renderer/inventory.js` (`_ttagMaterialRecords`, `_ttagCollectBackups`, `_buildTtag`, `_exportMaterialTtag`, `_exportSelectedTtag`, `_syncTtagExportBtn`, `_syncTtagBarButtons`)
- **`.ttag` import — validate → preview → accept** (Phase 1, the counterpart to export). The "Import" button (inventory toolbar `btnInvImportTtag`, next to Export / "+ Material"; and Settings → Data `btnStgImportTtag`) opens a **source modal** ("Import TigerData" — Shopify-CSV-import shape, minimalist: a big dashed drop zone whose "drag your .ttag files here" hint appears ONLY on hover, two source buttons, footer Cancel + primary, no header/footer hairlines). Sources are **staged as removable chips** — **Upload files** (multi-select picker), **Upload from URL** (a toggle revealing an http(s) field in RESERVED space so the buttons never shift; a deliberate user action, scheme-checked), and dropping onto the zone — then **Preview** reads/validates them all at once (a bad file/link is skipped-and-counted) and swaps to the preview screen. Plus **drag-and-drop a `.ttag` anywhere on the window** even without opening the modal (a full-window drop hint appears while a file is dragged; only reacts to real file drags, never the storage view's internal DnD, and stands down while the import modal is open so its own drop zone handles it). Once a source resolves, the same modal swaps to the preview. **Several `.ttag` at once** — the picker is multi-select and a drop can carry many files; each file is validated and keeps its **own** mode (a Restore file and an Import file can ride together → the batch reads as "mixed"), invalid files are skipped-and-counted (never fatal unless all fail), and every file's records merge into one preview and one batched write. The picker/drop → parses + **validates** the content marker (`format`/`kind`/`version ≤ 1`; rejects a non-`.ttag`, a newer version, or an empty file with a clear message) → **sanitises** every record (untrusted file: bad-scheme URL fields dropped via `_looksLikeUrl`, colour bytes/weights/TD clamped) → shows a **preview modal** (mode banner, materials count — the raw record count is not surfaced — and a **table** with column headers: a header **select-all checkbox** (all/some/none tri-state), a thumbnail column reusing the side-card spool illustration with a top-left twin badge, a material column, a **weight column (value + fill bar, like the inventory table) that live-switches to full 100% when Import is chosen**, and a tier-badge column; each row carries an **include checkbox (all on by default) — uncheck any material to leave it out of the import** (a twin is one material; the summary shows "X of N selected"; Accept disables at zero; the selection survives the Restore/Import mode switch); the list has no scrollbar — a soft bottom fade hints when it overflows). the preview lets the **user choose the add mode** (nothing decided by default) between two options, **pre-selected from the file's `exportedBy` as a hint** (same account → Restore, any other → Import) but freely switchable; Accept stays disabled until a mode is picked. **Restore** rewrites every record **verbatim** under its original id + restores each TigerTag+ `rfidList` backup (keeps the uid / TigerTag+ / backup); **Import** writes each record as a **fresh chipless material you own** (new `TigerData_` uid, fresh `id_tigertag` nonce so it's not Plus, `id_product` unset, `rfidBackup:false`, `twin_tag_uid` remapped through the new ids, backups dropped — a signed dump is bound to a chip you don't hold — and **weight reset to full capacity (100%)** so it lands as a brand-new spool). Both modes stamp a fresh `updatedAt` and never write `deleted`. Batched writes (chunked under Firestore's 500 cap), additive in import mode (never overwrites existing docs). New modal `#ttagImportOverlay` (reuses the add-friend card chrome), CSS in `60-modals.css` (rebuilt to use only real design tokens — `--surface`/`--surface-2`/`--border`/`--muted`/`--primary` — after an early pass wrongly referenced non-existent `--surface-1`/`--hairline`/`--faint` and rendered broken). - **USB scale (Dymo M-series) live weighing integration** — plug a Dymo USB scale and spool weights update themselves. Main process opens the HID device (vid `0x0922`, usage page `0x8D`, `node-hid` dep + electron-rebuild), decodes frames (status/unit/exponent, oz/lb→g), hot-plug poll every 5 s, streams `usb-scale-update` / `usb-scale-data` over IPC (+ `usb-scale:state` seed handle, preload bridge `onUsbScaleUpdate` / `onUsbScaleData` / `getUsbScaleState`). Renderer module `IoT/usbscale/` runs a one-write-per-pose state machine (≥50 g on scale starts a session; 3 consecutive identical stable frames ≈ 3 s = stabilised): **POD mode** — chip UID(s) on the RFID reader(s) identify the spool with certainty (side card open or not) → silent `doWeightUpdate(row, "raw", grams)` (gross→net via `container_weight`, twin synced in the same batch), cancelled if the UID set changes mid-weigh; **side-card mode** — no chip present but a detail panel open → confirmation card (spool puck + brand/material + gross→net math) applied in one click; missing container blocks the write and routes to the container picker. After a write/cancel the session latches: nothing rewrites until the chips leave the readers (POD) or the scale is emptied (side-card). The scale UI lives INSIDE the open side card's WEIGHT section (`#usbScaleDock` in `buildPanelHTML`): live grams + status row whenever the scale is connected, inline confirm block under it (clicks delegated on `#panelBody`; 1 Hz frames patch the grams node in place, structural sig guards rebuilds); a POD-mode container prompt auto-opens the target spool's side card. When a scale disconnects (the M-series auto-powers off after ~3 min idle) the dock shows a dimmed "asleep — tap to wake" hint, but ONLY after a scale connected at least once this session (`_everConnected` gate — no hint for users who never plugged one in). The scale block sits as its own borderless section below the WEIGHT card (pulled out for room; live readout = a real Dymo product photo `assets/img/spool_filament/dymo_usb_scale.png` at 50 px + grams/status stacked beside it). That same photo also replaces the illustration in the container-weight ("Edit Calibration") modal. ## Changed

- **The chipless `protocol` value is now `TigerData`, not `TigerCloud`** — `normalizeRow` computes `protocol: "TigerData"` for chipless (display, filter, and — via `syncSpoolMirrors` — the stored doc field all become `TigerData`), so the raw doc matches its `TigerData_` id. The cloud stat tile filters on `"TigerData"` too. Hub aligned (`lib/data/inventory.ts` fallback). **One value deliberately stays `TigerCloud`: the `byProtocol` telemetry rollup key** in the stats Cloud Function — it's derived from the id prefix, not the field, and keeping it stable avoids fragmenting historical analytics into two chipless buckets. Nothing cross-repo matches on the `"TigerCloud"` string (verified), so this is safe — `renderer/inventory.js`, Hub `lib/data/inventory.ts`, `llms.txt`
- **Chipless spool ids migrate silently from `CLOUD_` to `TigerData_` on login** — the counterpart to v2.13.1 minting new chipless ids as `TigerData_`. Done client-side, not server-side, on purpose: a server sweep would rename docs under clients that haven't updated to accept the `TigerData_` prefix yet (they'd read them as real chips), whereas the client only migrates accounts whose Studio already accepts both, spreading as users sign in. A chipless doc is self-contained (no twin, no `rfidList` backup, nothing references it by id), so each migration is a plain **atomic id-rename**: `create TigerData_<suffix>` (same data, `uid` field renamed, `updatedAt` preserved) + `delete CLOUD_<suffix>` in one Firestore batch. Idempotent (re-reads first; no-op when there are none), guarded against double-fire and safe vs a concurrent write. Runs in the inventory snapshot handler next to the existing decimal→hex and rack migrations — `renderer/inventory.js`

## Fixed
- **Rack view: can't scroll to off-screen racks while dragging a spool (Windows).** During an HTML5 drag, Windows/Chromium blocks the wheel and doesn't auto-scroll the inner rack column (`.rp-racks-scroll`) — so a Windows user holding a material couldn't reach racks below the fold (macOS scrolls natively, so it was invisible to us). Added a cursor-driven **edge auto-scroll**: nearing the top/bottom of the rack column (within 64px) scrolls it (16px/frame), started on drag and stopped on drop/dragend. **Gated to non-macOS** so it never stacks on macOS's native drag-scroll (which would double the speed). Wired once, active only while `body.is-dragging-spool`. — `renderer/inventory.js` (`_wireRackAutoScroll`)
- **Grouped-deck side-card: click-to-toggle + no auto-open during multi-select.** Clicking an inventory row/card opens the grouped deck and **clicking the same row again closes it** (`_toggleGroupPanelForSpool`). In multi-select mode the deck no longer pops open automatically on each pick — the **checkbox selects**, while **clicking the row/card body toggles the deck**. Grid view is now **iso with the table**: the per-card selection checkbox (revealed on hover, always shown in select mode) does the selecting, and a click on the card opens/toggles the grouped deck instead of selecting. — `renderer/inventory.js`, `renderer/css/70-detail-misc.css`
