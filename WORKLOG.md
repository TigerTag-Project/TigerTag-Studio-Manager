## Fixed
- Absolute-positioned icon buttons (password-eye on the printer-settings form `.pba-input-eye`, password-eye in the login modal `.lm-eye`, input-clear in the add-printer form `.adp-input-clear`) visibly jumped ~14 px down when clicked. The global `button:active { transform: translateY(1px); }` rule was overwriting each button's centering `transform: translateY(-50%)`. Each now has a dedicated `:active` override that re-applies the centering while keeping a 1 px press feedback — `renderer/css/40-printers.css`, `renderer/css/60-modals.css`
- Printer-settings `.pba-input` (Bambu access code, Creality password, every secret field) silently changed size when the eye toggle flipped `type="password"` → `type="text"`: text grew from 13 → 14 px and the box from 36 → 40 px. A global `input[type="text"]` rule (60-modals.css L1354, specificity 0,1,1) was outranking `.pba-input` (0,1,0), and that global rule doesn't match `input[type="password"]` — so the two modes rendered differently. Doubled-selector overrides (`.pba-input.pba-input`, specificity 0,2,0) re-assert height/padding/font-size; the field now stays exactly the same dimension whether the password is hidden or shown — `renderer/css/40-printers.css`
- **RFID rescan no longer wipes user-edited spool data, and the spool weight now syncs back onto the chip instead of being rolled back to a stale value.** The upsert previously did a full `set()` on the inventory doc on every rescan, erasing every user-edited Firestore field that isn't on the chip (container assignment, custom note, capacity, etc.) AND replacing the current `weight_available` with the chip's `measure_available_gr`. But the slider that updates the weight only writes to the DB — nothing ever writes it back to the chip — so the chip's value is almost always stale. Three-part fix: (1) switched to `set(..., { merge: true })` so user-edits survive; (2) weight is now only seeded from the chip on a **fresh write** (new chip, or after the intentional clean-slate delete on chip rewrite) — on a regular rescan the DB stays authoritative; (3) **auto-sync at scan**: when the DB weight differs from the chip's value, the app fires a surgical RFID write that rebuilds the payload from the Firestore doc and updates only page 0x17 (`Measure Available`, u24 BE) on the tag — fire-and-forget while the chip is still on the reader, so chip and DB converge automatically. New chips and chip-rewrite paths are unchanged — `renderer/inventory.js` (`_writeChipDoc` + RFID upsert loop)
