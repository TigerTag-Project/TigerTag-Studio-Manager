# Worklog ‚Äî v2.11.1

## Added
- **Xano chip-event analytics ‚Äî Studio now reports physical-chip events like the mobile app does.** Reads, writes and chip resets performed with a TigerPOD / NFC reader are reported to the TigerTag Xano analytics table (`api.tigertag.io/api:tigertag/analytics/tag`), so Studio and Tiger RFID Connect land in the **same table with the same shape**. Five operations, mirroring the mobile vocabulary: `read` and `online` (chip scanned + decoded ‚Äî see the routing note below), `write` (chip programmed ‚Äî fired per chip actually burned AND verified, where the mobile fires once per *batch* and so under-reports multi-chip writes), `reset` (Format ‚Üí back to the TigerTag Init state) and `nfc_init` (Recycle ‚Üí blanked to a plain NFC tag). Studio never emits `init` (mobile's "a foreign chip became a TigerTag") ‚Äî it has no such flow.
  - **A scan routes to one of TWO endpoints, and the choice decides whether the catalogue product is recorded at all.** A TigerTag+ whose chip carries a real catalogue product emits **`online`** to `product/get?uid=&product_id=` ‚Äî that endpoint takes `product_id` as a query input and hands it to the same `Analytics Header Data to DB` function, so the row lands WITH the product identity. Everything else (a maker chip, `write`, `reset`, `nfc_init`) goes to `analytics/tag`, which has **no `product_id` field at all** ‚Äî so those rows can never carry one, by design. Both endpoints read the same base64 `X-Tigertag-Analytics` header (it is literally how `product/get` detects an app call; the function aborts on a `DATA != null` precondition without it). Xano only records the `online` row when `product_id > 500`: ids below that are the reserved range for internal/demo chips, deliberately excluded from analytics. The `product/get` response (the product payload) is ignored ‚Äî Studio fetches that on its own path; Xano logs the analytics in a Post-Process block, i.e. after responding, so it costs the user nothing.
  - **Transport (not ours to change):** a **GET** whose JSON payload is base64-encoded into an `X-TigerTag-Analytics` header, on an unauthenticated endpoint. Sent from the **main process**, because the renderer is a browser context and a cross-origin GET carrying a custom header would trigger a CORS preflight the endpoint doesn't answer. Fire-and-forget by construction (`ipcMain.on`, not `handle`): a dropped event is lost, never retried, and can never block or slow a chip action ‚Äî same contract as mobile, so server-side counts are a lower bound.
  - **60 s dedup per (operation, UID)** ‚Äî the TigerPOD is a *resting* reader that re-reads the same chip on a loop and would otherwise flood the table. The mobile app needs no such guard (every scan there is a deliberate tap). A different operation on the same chip is never suppressed.
  - **UID sent in DECIMAL** (`BigInt("0x"+hex)`) ‚Äî Xano keys on decimal. Noted for the mobile team: the mobile's `online` operation puts the UID in **hex** into that same field (it reuses a mis-named `uidDecimal` variable), so those rows are mis-keyed in Xano.
  - **Privacy:** `Location` is always `"0.0,0.0"` ‚Äî Studio collects **no position**. A desktop doesn't move, so a precise coordinate would reveal the user's home address while answering no analytical question that `Country` (derived offline from the timezone ‚Äî no IP lookup, no permission prompt) doesn't already answer; Xano can still resolve the request IP on its side. `Emoji` is deprecated and always sent empty; `Message` is sent, as on mobile. A new `App: "studio"` field tells Studio rows apart from mobile ones, and `Platform` is sent as a product name (`macos` / `windows` / `linux`) rather than Node's kernel names ‚Äî `darwin` reads like noise next to the mobile's `ios`/`android`, and `win32` is actively misleading (Node returns it on 64-bit Windows too).
  ‚Äî `main.js` (`analytics:tag` IPC + the Xano request), `preload.js` (`sendTagAnalytics`), `renderer/inventory.js` (`_sendTagAnalytics` / `_xanoChipFields` / `_xanoUidDecimal`, wired into `_processNfcScans`, the encode/burn loop and `_wireChipReset`).
- **Server-side stats & history (backend repo) ‚Äî the foundation of the user + admin dashboards.** New in `TigerTag_Firebase_Backend`: `users/{uid}/stats/current` (rollup) and `users/{uid}/dataHistory/{autoId}` (time-series snapshots, ‚â§1 point/15 min) plus the global `adminStats/{YYYY-MM-DD}`. Computed **server-side only** ‚Äî a client-written business figure is untrustworthy (publicly connectable backend) and leaves holes in the curve whenever the app is closed. Architecture is *flag + reconciler, never a scan*: the `onDocumentWritten` triggers on inventory / products / printers / lists do nothing but set `dirty = true` (one write, zero reads ‚Äî a print job pushing a spool's weight down costs almost nothing), and a 15-min scheduled reconciler queries `collectionGroup("stats").where("dirty","==",true)` so it reads only the accounts that moved (dormant accounts are never billed). It clears the flag *before* recomputing (a write landing mid-recompute re-marks the account ‚Üí picked up next cycle; at worst one extra recompute, never a lost update) and recomputes the FULL snapshot rather than applying increments (an incremental counter that misses a delta drifts permanently and silently; a full recompute is self-healing). Raw points older than 90 days are downsampled to one per day, pruned lazily per account. Monetary values are always HT; the global rollup keeps `valueHtByCurrency` because summing different currencies would be meaningless. Also documented in `docs/firestore-schema.md` + the backend README. **Deployed** (rules, indexes, 7 functions) and backfilled over all 2 372 existing accounts; a Maintenance panel in the TigerHub admin console (`/admin`) can re-trigger the global rollup or a full rebuild on demand ‚Äî `firestore.rules`, `firestore.indexes.json`, `functions/index.js` in the backend repo.
- **`protocol` mirrored onto each inventory doc ‚Äî it's what makes the TigerTag / TigerTag+ split possible server-side.** Same reasoning as `productKey`: the protocol is resolved via `versionName(id_tigertag)` against the TigerTag reference DB, which the backend doesn't have. Counting the RAW `id_tigertag` server-side is not a substitute ‚Äî only a couple of values map to a known version and the rest are a long tail of unrecognised ids, so it produced ~200 noise keys *and* no TigerTag-vs-TigerTag+ split at all (the Hub could only show "chip vs TigerCloud"). The client now mirrors the resolved string (`TigerTag` / `TigerTag+` / `TigerCloud` / `unknown`) and the backend counts a clean `byProtocol` map, replacing the raw `byVersionId` ‚Äî `renderer/inventory.js` (`syncSpoolMirrors`, renamed from `syncProductKeys` now that it mirrors two fields), backend `functions/index.js` (`computeUserStats`, `rollupGlobalStats`, the inventory trigger's changed-field list).
- **`productKey` mirrored onto each inventory doc ‚Äî the join key for the upcoming server-side stats.** The product identity hash (`_productKeyHash`) is built from RESOLVED names (brand / material / aspect come from the TigerTag reference DB), so a backend job cannot recompute it from the raw Firestore fields ‚Äî it could never join a spool to its `products/{keyHash}` record and the stock value would come out as 0. The hash is now denormalised onto `users/{uid}/inventory/{spoolId}.productKey` (same pattern as the `rfidBackup` mirror flag), so the backend does a plain lookup: no reference DB server-side, no second copy of the hashing logic, no drift when the catalogue changes. Written from the inventory snapshot handler (`syncProductKeys`) rather than the ~10 spool-write sites ‚Äî one self-healing path covering new spools, identity edits (colour/brand change ‚Üí new hash) and the one-time backfill of existing spools; it only writes when the stored value actually differs, so the write settles on the next snapshot instead of looping, and it bails until the reference DB has loaded (otherwise rows would hash from fallback labels and persist a wrong key). No rules change needed (`inventory` has no `hasOnly` whitelist) ‚Äî `renderer/inventory.js` (`syncProductKeys`, `subscribeInventory`).
- **New `cart-plus` icon** (`assets/svg/icons/icon_cart_plus.svg`, `.icon-cart-plus` mask) ‚Äî a cart with a **Ôºã** inside the basket, drawn in the same 24-viewBox / 2px round-stroke style as the existing cart.

## Fixed
- **Twin Tiger Tag spool on the Pod: the side-card opened, then swapped to a second card showing "reset" data.** A twin spool = TWO Firestore docs (one per chip UID), independent by design. The per-spool custom fields (custom image, container) were saved on only ONE of the two docs, leaving the twin "hollow"; when a scan opened the card on the hollow chip, it looked reset. (Invisible on Tiger Tag+ ‚Äî both docs render the same shared catalogue data.) Two-part fix:
  - **Root cause ‚Äî mirror the remaining per-spool fields onto the twin.** Weight, rack placement and tags already mirrored at their write sites; added the same for **custom image** and **container** (id + weight) via a shared `_updateSpoolTwinned(r, patch)` helper, so both chips of one physical spool now always carry the same custom data. This also fixes the grid/stats, which read the raw docs.
  - **Symptom ‚Äî stop the card swap.** The pod's two readers fire a bare `rfid-uid` each ~ms apart; before the twin link resolves (~1.5 s later, in `_processNfcScans`) the anti-duplicate guard couldn't tell the two chips were one spool, so the second swapped the card to its own doc. Widened the guard to also treat the scan as the same spool when the scanned chip shares the open card's product identity (`_spoolGroupKey`, available on both docs immediately) ‚Äî gated to the twin-arrival burst (< 1.6 s of the card being scan-opened) so scanning ANOTHER physical spool of the same product later still opens its own card. No added delay; instant open preserved.
  - **Existing data ‚Äî a one-shot repair (`reconcileTwinFields`).** New spools mirror going forward, but twins configured before this fix stay inconsistent. A reconciler fills each twin's hollow chip from its configured half across image / container / rack / tags / note ‚Äî **fill-gaps only** (copies a field only from the twin that HAS it to the one that doesn't; never overwrites a present value, so no data loss and no "which is right?" guess; a genuine both-present conflict is logged, not touched). Deliberately NOT real-time: the localStorage-gated check runs only on the FIRST server snapshot of a session and the actual reconciliation runs **once ever per account**; it never re-runs on later snapshots. An admin **Debug ‚Üí "Repair twin spools"** button forces it again (reads the in-memory inventory, writes only where a gap exists ‚Äî the changed fields aren't in the stats trigger's watch list, so no backend churn). The last run's outcome (pairs checked / docs filled + timestamp) is persisted and shown in the Debug panel, so the silent first-launch pass is visible after the fact.
  ‚Äî `renderer/inventory.js` (`_updateSpoolTwinned` + custom-image / container write sites, `_consumeScanOpen`, `reconcileTwinFields` + one-shot gating), `renderer/inventory.html` (debug-panel Tools button).
- **A friend's "Stock value" was far too low ‚Äî it only counted the products you happened to own too.** `_getProduct(r)` always resolves against `state.products` (MY records), but in friend-view the rows on screen are the FRIEND's spools. So the value loop (`price √ó remaining-weight fraction`) skipped every spool for which I had no priced product of my own (`if (!Number.isFinite(price)) continue`), collapsing the total to the *intersection* of our two shelves ‚Äî likewise the price column and price sorting showed "-" for the friend's spools. Added `_displayProduct(r)`, which resolves the product record DESCRIBING the row on screen (`state.friendProducts` in friend-view, `state.products` otherwise), and switched the six "displayed price" sites to it: the stock-value stat, price sorting (rows + groups), the table price cell, and the row/group render signatures. `_getProduct()` stays MINE everywhere it must (‚ù§/‚òÖ toggles, min stock, note, tags, bulk flags, the owner-only telemetry snapshot) ‚Äî the toggles write to my own account even while browsing a friend. Second half of the bug: the friend-products snapshot refreshed only the open card and the favorites view, so when it landed *after* the inventory render the value stayed stuck at its pre-products figure ‚Äî it now re-runs `renderStats()` and the inventory render too ‚Äî `renderer/inventory.js` (`_displayProduct`, `renderStats`, `_priceCell`, `_rowSignature`, `_groupHeaderSig`, `subscribeFriendProducts`).
