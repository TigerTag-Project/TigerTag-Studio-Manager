# Worklog — v1.10.30 (in progress)

## Added
- Toolbox: device-gated tools now stay visible but greyed (desaturated + dimmed via `.toolbox-row--greyed`, not `disabled`) instead of being hidden, so the toolbox is stable whether the hardware is plugged in or not. Chip features (Encode Cloud / Burn RFID / Restore TigerTag+ / Erase / Recycle) are ALWAYS shown for a spool in one of three states: FUNCTIONAL (reader + the right chip present), LOCKED (no reader at all → greyed, click opens the "TigerPOD Free STL" promo via `.toolbox-row--pod-lock` → `openTigerPodModal`), or GREYED-inert (a reader IS connected but this tool's chip/card isn't there yet → greyed with no POD ad, since they already own the hardware; placing the chip flips it to functional live). The `*Locked` id variants carry no real wiring so functional handlers (by id) never fire on a greyed row; mirrors each feature's real conditions (Cloud → Encode; non-Cloud → Burn / Restore-if-backup / Erase / Recycle), skipped in friend view. Twin-tagged spools: erase / recycle / restore / burn light up only when BOTH the UID and the twin UID are on readers (they act on the whole pair) — `_allChipsPresent` requires every one of the spool's chips present, and `_panelCardSig` counts them (0/1/2) so placing the second twin chip flips the tools live. Panel now refreshes on reader plug/unplug too (`_panelCardSig` includes reader count; `onRfidReaderUpdate` → `_maybeRefreshPanelForCard`) so the states flip live. TD1S measure tools ("Scanner la couleur" / "Scanner le TD") appear greyed when the TD1S sensor isn't connected — behaviour unchanged (the click already opens the "Capteur TD1S" connect modal), only the visual signals the block; for the split TD row only the main button is greyed so the trailing "clear TD" stays usable — `renderer/inventory.js` (toolbox chip-tools builder, `greyed` on measure tools, `_panelCardSig`, POD wiring), `renderer/css/70-detail-misc.css` (`.toolbox-row--greyed`)

## Changed

No more image reload / video restart / scroll jump on a card-presence change — `renderer/inventory.js` (`_wireToolbox`, `_maybeRefreshPanelForCard`)

## Fixed
- Windows: RFID readers (ACR122U / TigerPOD) plugged in AFTER the app started are now detected without a restart. Root cause: Windows' Smart Card service (SCardSvr) is trigger-started — it stops when no reader is present, and pcsclite's reader-monitor errors out and never re-arms, so a later hot-plug went unseen. Added a win32-only watchdog to `services/nfc-process.js`: on a context-level error it tears down and re-creates the `NFC` instance on a 3 s timer, retrying until a reader appears (which restarts SCardSvr); a fresh reader connection cancels the retry. macOS/Linux (persistent pcscd) path is unchanged (`_autoRecover` gated to win32) — `services/nfc-process.js`
- Opening any side card (spool / printer / friends / settings / debug / explorer / group / add-printer) now closes the notifications panel, so they never overlap — `closeNotifs()` added to `_closeAllSidePanels` and to each top-level open function — `renderer/inventory.js`
- Presenting a SINGLE chip of a twin pair no longer erases the twin link. `_writeChipDoc` wrote `twin_tag_uid: twinUid || null` with `{merge:true}`, so a one-chip scan (twinUid null) overwrote the existing pairing with null — after which the two chips were treated as unrelated (e.g. scanning the second chip re-opened a detail card over the already-open one). Now the twin id is only written when a partner is actually detected; a partial scan preserves the link (un-linking stays an explicit action) — `renderer/inventory.js` (`_writeChipDoc`)
- Re-presenting a chip no longer restarts/reloads the material video in the open detail card. Both panel-refresh paths that rebuild the card (chip-presence toolbox flip `_maybeRefreshPanelForCard`, and structural snapshot diff `refreshOpenDetail` path 3) now go through `_rebuildDetailKeepVideo`, which reuses the existing `<video>` node (a detached media element keeps its currentTime + buffer) instead of the freshly-built one — so the clip keeps playing across the rebuild — `renderer/inventory.js`
- Rescanning an already-open spool no longer pops a second detail card on top. The guard in `_consumeScanOpen` now tests the scanned UID directly against the OPEN spool's own fields (uid / twinUid / spoolId) instead of a freshly-looked-up row whose twin link can be momentarily stale right after a rescan write — so scanning either chip of a twin pair (main UID or twin UID) while its card is open does nothing, and it also requires the panel to actually be `.open` — `renderer/inventory.js` (`_consumeScanOpen`)
