# Worklog — v1.10.7 (in progress)

## Added
- Bambu Lab machine-control card in the printer side-card, matching the other brands (Anycubic-style jog pad). Play/pause/stop (in the job card), XYZ jog + per-axis/all homing, chamber/work light toggle, disable motors, cooling fans (part `M106 P1`, auxiliary "assist" `M106 P2`, and chamber/"case" `M106 P3` on enclosed models like the X1C — each toggle + ±10%), nozzle/bed temperature setpoints (click-to-edit pills, preheat at idle), and print-speed level (Silent/Standard/Sport/Extreme). Pause/resume/stop use the documented MQTT commands; light uses `system.ledctrl`, the rest use `print.gcode_line`/`print.print_speed` (OpenBambuAPI payloads — **to verify per model on real hardware**, A1 has no chamber light). Live state parsed back from telemetry: light (`lights_report`), speed level (`spd_lvl`). Fans: Bambu only reports actual, fluctuating speed (no setpoint field), so the UI accepts a telemetry value only when it repeats **two reports in a row** (stable) — filters the ramp's transient values while still catching a change made on the printer. User commands set the value optimistically and reset the confirm tracker. No `main.js` change — reuses the existing `bambulab:publish` IPC, so reload-only. `printers/bambulab/index.js`, `printers/bambulab/cards.js`, `printers/bambulab/PROTOCOL.md` (§5.6), `inventory.js`

## Changed
- Printer config (add/edit form, `#printerAddPanel`) now behaves like the spool sidecard: **non-modal** (no dimming overlay — the list + printer panel stay visible/usable) and, when opened from a printer's side-panel (gear), it tucks **to the left of the printer panel** via `_syncPanels` instead of opening hidden behind it. In the add flow (no printer panel open) it still opens at the right edge. Gets the same **`»` close tab** as the spool/printer side-cards (glued-slide via `_setTab`, z-index 101 so it passes behind the printer panel), in addition to the header ✕ / Back. Switching to a different printer's side-card auto-closes a stale settings panel. `renderer/inventory.js`, `inventory.html`, `css/70-detail-misc.css`
- FlashForge discovery now obtains the serial number automatically (slicer-parity). Modern models (Creator 5 / 5 Pro, 5M, AD5X) answer a UDP datagram on port **19000** (unicast or broadcast) with a credential-free identity record (model name + serial) — the same mechanism FlashPrint / Orca-FlashForge use before asking for only the access code. The serial NEVER appears in the HTTP `/detail` payload, so this was the missing piece. New main-process IPC `ffg:udp-probe` (parser `_ffgParseUdpIdentity`, 280-byte fixed layout: model @0, serial after an 18-byte header), preload bridge `ffgUdpProbe`, and `ffgProbeIp` now enriches via UDP first (TCP M115 stays as the legacy fallback). Result: scan + Add-by-IP fully identify the printer (model + serial) and the user only needs to enter the access code. Confirmed against a real Creator 5 Pro (firmware 1.9.2, SN parsed correctly). `main.js`, `preload.js`, `printers/flashforge/probe.js`, `printers/flashforge/PROTOCOL.md` (§2.7), `CODEMAP-main.md`
- FlashForge model is now auto-detected on first connection. Newer models (Creator 5 / 5 Pro) hide their identity from the unauthenticated add-by-IP probe, so they get added as "Select Printer" with the placeholder image. The live driver now reads `detail.model` on the first authenticated `/detail`, resolves it (`ffgModelIdFromMachineModel`), and persists the corrected `printerModelId` to Firestore once — so the right name + image appear automatically without the user picking the model by hand. Generic bridge `ctx.updatePrinterModel` → `persistPrinterModel` (writes `printers/{brand}/devices/{id}`). `printers/flashforge/index.js`, `printers/context.js`, `inventory.js`
- FlashForge Creator 5 Pro is now modelled as a **tool-changer** (like the Snapmaker U1) instead of the AD5X "one nozzle + CFS multiplexer" layout. Its `/detail` exposes a per-nozzle temperature array whose length matches the slot count (`nozzleTemps.length === slotCnt > 1`), which the live parser now uses as the tool-changer fingerprint: each station slot is bound 1:1 to its own nozzle/spool, rendered as tools `T1…T4` (no "Ext." entry) with each tool showing its own hotend temperature, and the head's currently selected nozzle (`currentSlot`) flagged active. The global nozzle pill in the Temperature block is hidden for tool-changers (per-tool temps replace it); bed + chamber remain. Material assignment still dispatches `msConfig_cmd` slot 1-N. AD5X / 5M (single nozzle) are unaffected. Confirmed against a real Creator 5 Pro (firmware 1.9.2). `printers/flashforge/index.js`, `printers/flashforge/cards.js`, `css/50-snapmaker.css`
- FlashForge discovery is no longer blocked by newer models. The scan + Add-by-IP + manual probe used to drop any printer that answered `/detail` on 8898 without an identity field — newer firmware (e.g. Creator 5) replies `{"code":1,"message":"SN is different"}` to the empty-SN probe and exposes no TCP 8899 to enrich from, so the probe returned `null` and showed "No FlashForge printer responded". Now an IP whose reply has the FlashForge shape but no identity is surfaced as an *unidentified* candidate (modelId "0"): the scan lists it and Add-by-IP opens Printer Settings prefilled so the user can pick the model and enter the SN + access code. `printers/flashforge/probe.js`, `printers/flashforge/add-flow.js`
- FlashForge model resolution now recognises `Creator 5` → id "5" and `Creator 5 Pro` → id "6" (catalog entries already existed). `printers/flashforge/probe.js`

## Fixed
- Bambu AMS humidity/temperature display was wrong. (1) AMS **Lite** (e.g. A1) has no humidity/temperature sensor but the card showed defaults ("5% · 0.0°C") — now shows nothing. (2) The humidity % used the `humidity` field, which is a **1-5 desiccant grade**, not a percentage (X1C AMS HT showed "1%" instead of the real 51%) — now reads the real `humidity_raw` (AMS 2 Pro / HT). Temperature is shown only when it's a real reading (> 0). AMS type is detected from the `get_version` `module[]` name prefix (`ams_f1/`=Lite, `n3s/`=HT, `n3f/`=2 Pro, `ams/`=standard) per PROTOCOL §8.3, and AMS Lite is explicitly hidden. `printers/bambulab/index.js`, `printers/bambulab/cards.js`
- FlashForge temperature card: the chamber pill (FlashForge "Case" temp, `chamberTemp`) used the bed icon — now uses the chamber/enclosure icon (`SNAP_ICON_CHAMBER`). `printers/flashforge/cards.js`
- FlashForge CFS slots no longer vanish on the Creator 5 Pro. Its `/detail` ships `matlStationInfo` (slotCnt + slotInfos) but omits the `hasMatlStation` flag that AD5X/5M set, so the live parser fell through to the single-extruder fallback and dropped all 4 station slots. Material-station detection now also triggers on a populated `matlStationInfo` (non-empty `slotInfos` or `slotCnt > 0`). Confirmed against a real Creator 5 Pro (firmware 1.9.2). `printers/flashforge/index.js`
- FlashForge catalog: Creator 5 Pro pointed at `a5.png`; now uses the existing `creator_5_pro.png`. `data/printers/ffg_printer_models.json`
- Live printer cards no longer rebuild from scratch on every telemetry message. Added a shared in-place DOM reconciler (`printers/dom-morph.js`, morphdom-style) that diffs the freshly-rendered HTML against the live DOM and mutates only what changed — so an incoming MQTT report can no longer close an open inline edit / `<select>`, focus & scroll survive, and there's no flicker. Only what the user is actively *editing* is left untouched (focused input/select/textarea/contenteditable, or `.snap-temp--editing`) — a merely-focused `<button>` does NOT freeze its card (else clicking e.g. the LED kept focus on it and the card stopped refreshing until the next click elsewhere). Optimistic-state commands (light/fan/speed) repaint immediately via `_bblNotify` instead of waiting for the next report. Wired on Bambu Lab (`renderBambuLiveInner` + log via `morphInner`); the other brands are a one-line swap to follow. `printers/dom-morph.js`, `printers/bambulab/index.js`

## Docs
- FlashForge PROTOCOL.md §13 corrected: `/control` `msConfig_cmd`/`ipdMsConfig_cmd` `rgb` is sent **WITH** the `#` prefix (`"#FF5733"`), not bare hex — verified by capturing the FlashForge slicer's own request. Documented the Creator 5 Pro colour constraint: tested over the network (slicer closed, no RFID/scanner hardware), only certain `rgb` values "stick" (those picked via the slicer, e.g. `#F82D29`/`#FFF245`); any arbitrary colour reverts to `#FFFFFF` while the printer still ACKs `code:0`. Likely a firmware palette restriction (FlashForge official filament colours); not a payload bug. App code already sent `#RRGGBB` (`ffgColorToHash`) — confirmed correct, left unchanged. `printers/flashforge/PROTOCOL.md`
