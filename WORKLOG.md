Updates live with the job (same surgical `_patchTableJobs` path) — `renderer/inventory.js` (`_fmtEndClock`/`_etaSortVal`, sortable pt-head + pt-row eta cell), `renderer/css/40-printers.css` (`.pt-td--eta`), i18n `printerEndsAt`

## Fixed
- The Snapmaker "Recommended firmware" wizard (Paxx U1 Extended Firmware — the 3 steps: install firmware, USB update, enable TigerTag reading) was hardcoded in English. It's now fully translated into the 9 locales (product names Paxx / OpenRFID / Buy Me a Coffee kept as-is) — `renderer/printers/snapmaker/index.js` (all wizard strings via `t()`), i18n `snapPaxxStep1Title`/`snapPaxxStep1Desc`/`snapPaxxDownload`/`snapPaxxViewRelease`/`snapPaxxWatchExplainer`/`snapPaxxCoffeeTitle`/`snapPaxxStep2Title`/`snapPaxxStep2Sub`/`snapPaxxStep2Li1..3`/`snapPaxxStep3Title`/`snapPaxxStep3Sub`/`snapPaxxStep3Desc`/`snapPaxxShowScreenshot`/`snapPaxxTutoTitle`/`snapPaxxOpenConfig`/`snapPaxxDocs`
- Editing a filament's colour then "Update RFID" left the chip's colour unchanged (a rescan still showed the old colour). Root cause: the displayed/edited colour lives in `online_color_list` (hex), but the physical chip colour is written from the RGB fields `color_r/g/b` (+ 2/3) via SDK `fromCloudDoc().toBytes()` — and the two were never synced, so the surgical write saw the colour page unchanged and only rewrote the timestamp page ("1 page written + verified" — looked OK but the colour never changed). Fixed on both sides: (1) the chip write (`main.js` `rfid:encode-cloud`) now treats `online_color_list` as the source of truth and patches the tag's colour bytes from it before serialising — so an update re-writes the new colour even when the doc's `color_r/g/b` lag; (2) the colour save (`renderer/IoT/td1s/edit-modals.js` `_saveTdHex`) also mirrors the edited hex list into `color_r/g/b`/`color_r2..3` so the doc stays consistent for rescans. Confirmed on real hardware (chip colour now updates + reads back)
- Changing a filament's colour didn't refresh its illustration in the "Spools not stored" panel until a later rebuild. The rack view's surgical patch (`_tryPatchRackSlots`) only re-rendered rack *slots*, never the not-stored side-rows — and a colour edit leaves rack membership unchanged, so the patch path was taken and the side panel stayed stale. The rack snapshot now also captures each not-stored spool's rendered inner HTML (`unrankedFill`), and the patch refreshes any `.rp-side-row` whose rendering changed, in place (keeps the row's drag/click listeners) — `renderer/inventory.js` (`_computeRackSnapshot` `unrankedFill`, `_tryPatchRackSlots` side-row patch)
- Clicking a material's "Storage location" row jumps to the Storage view but no longer highlighted its slot. Root cause (found by tracing the live console): `renderRackView`'s no-op guard (`if (_sig === _lastRackViewSig && _built) return;`, added for anti-flicker) early-returns when the rack structure is unchanged — which is exactly the locate case (the DOM doesn't change, only `state.locateUid` does) — so it skipped `applyRackSearchDim` entirely and the slot never lit up. Fixed by re-applying `applyRackSearchDim()` on the existing slots right before that early return; also covers the same gap for the search box / quick-filters when the rack view is already on screen. The surgical patch path (`_tryPatchRackSlots`) got the matching `state.locateUid` + filters guard too — `renderer/inventory.js` (`renderRackView` no-op-guard re-apply + `_tryPatchRackSlots` guard)
- Elegoo print progress % jumping wildly (e.g. 100% ↔ 10% ↔ 31%). Root cause (confirmed by capturing the live MQTT stream from a real Centauri print): the firmware emits three conflicting progress signals that overwrote each other — (a) `api_status` (6000) pushes carrying `print_status.print_duration` but NO `remaining_time_sec` key, which the parser read as `elapsed/(elapsed+0)` = 100%; (b) `1005` time-based estimates; (c) the real `machine_status.progress`. Fix in `renderer/printers/elegoo/index.js` (`_mergeStatus`): `machine_status.progress` (the firmware's own 0-100 value, what the printer screen shows) is now the authoritative source and locks out the noisy estimate for the rest of the job; the time estimate is a last-resort fallback used only when `remaining_time_sec` is present AND > 0 (kills the 100% spikes and the stale-idle 100%); and `machine_status.progress` is scaled /100 unconditionally (the old ">1 → /100" heuristic misread progress==1 as 100% and froze the bar there through heat-up). Replaying the fix over the real log gives a clean monotonic 0→86%. Documented the whole progress source-of-truth + pitfalls in `renderer/printers/elegoo/PROTOCOL.md` (new §21.1 + corrected §6: `machine_status.progress` is 0-100/÷100 and pushed ~2s, not 0-1/~15s).
- Search bar now filters the printer view (grid + table) — it previously only worked for materials/racks, so typing did nothing in the Printers view. Matches on printer name, brand, model, IP / broker / "cloud" and tags; filters in place (toggles `.hidden`, collapses empty online/offline grid sections) without rebuilding — so it never re-fires brand connects/pings on each keystroke. Cam wall keeps its live feeds unfiltered. The search placeholder now adapts per view ("Search printer, brand, model, IP…" in the printer view), and the right-of-search selectors are repurposed for printers: Brand (printer brands), a State filter (online/offline — the materials Material select repurposed), and Tags (printer tags), all scoped to printers with separate state so materials filters aren't touched; the materials-only Version select is hidden and the Tag select hides itself when no printer has a tag — `renderer/inventory.js` (`_printerMatchesSearch`, `applyPrinterFilter` incl. brand+state+tag, `populatePrinterFilters`, context-aware brand/material(state)/tag `change` handlers, `_onFilterChange` printer branch, `_syncSearchPlaceholder`, `printer-filters` body class in `setViewMode`, re-localise on language switch), `renderer/css/40-printers.css` (hide Version), i18n `searchPlaceholderPrinter`, `filterAllStates`
