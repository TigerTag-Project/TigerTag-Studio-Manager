# Worklog — v1.8.20 (in progress)

## Added
- **Anycubic printer support (6th brand)** — full LAN integration with the ACE (multi-color box): live box/slot layout in the printer side panel, slot filament editing (base type + RGB), online badge, request log, add-flow and Firestore persistence (`users/{uid}/printers/anycubic/devices`). Protocol: MQTT TLS 1.2 directly to the printer's own broker (`mqtts://<ip>:9883`, self-signed cert, username/password) using the `multiColorBox` getInfo/setInfo channel reverse-engineered in the ACE-RFID project — full reference distilled into `renderer/printers/anycubic/PROTOCOL.md`. New brand folder `renderer/printers/anycubic/` (`index.js` driver + lazy filament-edit sheet, `cards.js` ACE slot card, `probe.js`, `add-flow.js`, `settings.js`, `widget_camera.js` placeholder), `data/printers/acu_printer_models.json` (Kobra 3 / 3 V2 / 3 Max / S1 / X), `assets/img/acu_printers/` (placeholder images — **real product photos still to be added**, filenames already referenced by the catalog), brand wiring across `renderer/inventory.js`, MQTT bridge + IPC in `main.js`, `window.anycubic` bridge in `preload.js`.
  - **Credential provisioning is slicer-import, not user-typed**: the broker username/password/deviceId/modelId are cloud-issued per pairing and unknowable to the user. The add flow's primary path "Import from Anycubic Slicer" reads every paired LAN printer's durable credentials from AnycubicSlicerNext's on-disk config (`machine_list_of_LAN`, keyless base64/+5 obfuscation decoded in `main.js` — slicer does NOT need to run). Scan (TCP :18910 + `GET /info`) and manual-IP paths merge with the imported credentials by IP and can repair a stale DHCP IP.
  - Known limits (by protocol, documented in PROTOCOL.md): printers must be in LAN mode; first-time pairing always happens in the slicer; `setInfo` carries only base type + color (full profiles need an Anycubic RFID tag).
- **Anycubic live telemetry + camera (phase 2)** — the MQTT bridge now subscribes to the printer's whole public report subtree, and the driver parses the `print` / `tempature` / `fan` / `status` report families (field shapes cross-checked against the open-source hass-anycubic_cloud integration; documented in PROTOCOL.md §5b): the side panel gains a **print-job card** (filename, %, remaining time, layers, state pill) and a **temperature card** (nozzle + bed), and Anycubic cards show live job progress in the printers grid/table (`_getPrinterJob`). **Camera**: the printer's HTTP-FLV stream (port 18088) is remuxed to ~5 fps JPEG frames by ffmpeg in the main process (`anycubic:cam-start/stop` + `anycubic:cam-frame`, same pattern as the Bambu RTSP cam) and displayed in the side panel, the cam wall, and the detached camera window (new `acu_ipc` cam type). Background auto-connects skip the camera (`skipCam`) so ffmpeg only runs while a panel shows the feed — `main.js` (shared ffmpeg detection hoisted out of the Bambu block), `preload.js`, `renderer/printers/anycubic/{index,cards,widget_camera}.js`, `renderer/printers/anycubic/anycubic.css` (new, linked in `inventory.html`), `renderer/inventory.js`, `renderer/cam/{cam.js,cam-preload.js}`.
  - **Camera is probe-gated (the FLV stream is on-demand).** Hardware testing showed `/flv` returns 404 until the printer is told to start capturing. Added `anycubic:flv-probe` IPC + `flvProbe`/`onCamEnded` bridge; the driver probes `/flv` before spawning ffmpeg and only shows the feed when a stream is actually live — no more 404 retry-loop spam, no wasted ffmpeg. ffmpeg never starts from background grid/table connects (`skipCam`). Kobra X model id `20030` mapped in `probe.js` + PROTOCOL.md §6.
  - **Standalone camera activation (captured + reproduced).** Sniffed the local broker (`scripts/acu-mqtt-sniff.mjs`, new dev tool) and captured the slicer's camera-start command: publish `{type:"video",action:"startCapture"}` to the `…/web/printer/{model}/{device}/video` topic; the printer confirms with a `video/report` `state:"initSuccess"` and `/flv` goes live. The driver now **activates the stream itself** — no slicer needed: on panel/cam-wall open it sends `startCapture` once MQTT is up, attaches ffmpeg on the `initSuccess` report (bounded `flvProbe` retry covers the race), and sends `stopCapture` + stops ffmpeg on panel close (`acuReleaseCamera`) or disconnect. It also reacts to `video/report` from any origin, so a stream started in the slicer attaches too; if the printer times the FLV out (`cam-ended`) it re-requests capture. Generalized the request builder + `anycubic:publish` to carry the `video` endpoint. Probe window widened to ~15 s (the printer can take ~6 s to start serving `/flv` after `initSuccess`).
  - **Two camera transports — gated on `/info.rtspUrl`.** Hardware testing found two camera types: **local HTTP-FLV** (Kobra 3 V2 — `rtspUrl` present, only `initSuccess`, real FLV) which is fully supported, and **WebRTC/TRTC** (Kobra X 20030 — no `rtspUrl`, reports `initSuccess`→`joinSuccess`→`pushStarted`, `/flv` 400s) which is the cloud-relay path and **out of scope**. The driver now checks `/info` for an `rtspUrl` (cached as `conn.data.camSupported`) before activating: WebRTC models stay on the hero photo, are never sent `startCapture`, and aren't probe-spammed. Added `scripts/acu-cam-test.mjs` (dev tool: activates + polls /flv to classify a model's camera).
- Printer connection tutorial pill in the **printer settings panel** (Add Printer manual form + Edit Printer panel) for brands shipping a `tutorial.json` (Bambu Lab, FlashForge, Elegoo) — a "📖 Tutoriel de connexion" button at the top of the form that opens the multi-step tutorial without leaving the configuration flow. Routes through the existing global `[data-printer-tuto]` delegate so click → `openPrinterTutorial(brand, modelName)`. Hint is filled in **dynamically** by reading `#pbaMpTriggerText.textContent` (live model name from the picker) at click time, so changing the model dropdown changes which tutorial opens — a Bambu user who picks "X1 Carbon" lands on the X1/H2/P2 series, "P1S" lands on P1, "A1 mini" lands on A1. Picking the placeholder model opens the full model picker — `renderer/printers/modal-helpers.js`, `renderer/css/40-printers.css`.
- FlashForge scan result card shows the **printer model thumbnail** on the left (56×56), matching the Flutter mobile app's scan tile. Pulls the image from `data/printers/ffg_printer_models.json` via `ctx.printerImageUrlFor("flashforge", modelId)`; falls back to the "Select Printer" placeholder when the modelId can't be resolved — `renderer/printers/flashforge/add-flow.js`.
- Bambu Lab scan modal now has the "Autres réseaux à scanner" block (input + Add button + chips), matching the 4 unicast brand modals. Wires into the shared `printers/extra-subnets.js` store — adding a prefix from the Bambu modal makes it visible immediately in any other open brand scan modal and persisted to `users/{uid}.scanExtraSubnets` in Firestore. Bambu's own SSDP-multicast scanner doesn't honour /24 prefixes (link-local by design), but the UI is kept here for consistency so the user can manage the shared list from any brand modal — `renderer/printers/bambulab/add-flow.js`.

- **Anycubic cloud-mode printer support (phase 3)** — printers in **cloud mode** (no local ports — e.g. a Kobra X, or any printer not switched to LAN) are now reachable through Anycubic's cloud, coexisting with LAN printers in one list (mixed fleet). Ported from ACE-RFID's `CloudApi`: signed REST (`getPrinters`/`sendOrder` 1206 getInfo / 1211 setSlot) + a shared cloud-MQTT connection (bundled client cert + RSA-token creds, TLS 1.2 `SECLEVEL=0`) that streams the same report families the LAN path parses (`_acuMerge` reused → live slots/job/temps). Set-filament works over cloud (REST sendOrder). New `services/anycubicCloudCerts.js` (bundled certs), `main.js` cloud bridge (`anycubic:cloud-*` IPC), `window.anycubic.cloud` in `preload.js`, `mode:"cloud"` branch in `renderer/printers/anycubic/index.js`, "Add a cloud printer" panel in `add-flow.js`, `ctx.addAnycubicCloudPrinter` + cloud-aware connect guards + cloud-edit guard in `inventory.js`, `anycubic.css` help block, 12 i18n keys ×9 locales. Validated end-to-end on a cloud Kobra 3 V2 (`scripts/acu-cloud-test.mjs`).
  - **Token acquisition is attach-only CDP (no slicer auto-launch).** The workbench session token isn't persisted anywhere on disk (verified: config holds only an OAuth token the workbench rejects; WebView2 stores it as `null`). So provisioning attaches over CDP to a **user-run** bridge-mode slicer (`--remote-debugging-port=9222`, signed in, Workbench open) and reads `GET_TOKEN`/`GET_USER_INFO` — the app never launches the slicer. The token expires → re-run "Add a cloud printer". Cloud printers have **no camera** (Kobra X camera is TRTC/cloud — out of scope). LAN mode is completely unchanged.
  - Dev tools added: `scripts/acu-cloud-test.mjs` (CDP→getPrinters→cloud-MQTT validator). The dead on-disk OAuth-token IPC was removed (it read a token the workbench rejects); LAN provisioning (`anycubic:read-slicer-config`) untouched.
  - **BoringSSL fix:** Electron's main process is BoringSSL, which (a) rejects the OpenSSL `@SECLEVEL=0` cipher string (`INVALID_COMMAND`) and (b) can't parse the bundled legacy (SHA1/RC2) PKCS#12. So the cloud MQTT client now uses the cert+key as **PEM** (extracted into `anycubicCloudCerts.js` as `CLIENT_CERT_PEM`/`CLIENT_KEY_PEM`) with no cipher tweak — BoringSSL has no SECLEVEL gate (the slicer itself is Chromium/BoringSSL on this broker). Connect wrapped in try/catch so a TLS error reports as cloud-status instead of crashing.
  - Printer card connection label is mode-aware: cloud printers show **"Cloud"** instead of "MQTT (LAN)" (`printerConnCloud` i18n key, 9 locales).
- **Fixed** — Anycubic ACE filament card dropped slots: the "external" box (`id -1`) is itself a **multi-slot** unit (a Kobra X / ACE Pro 2 reports box -1 with **4** slots), but the card collapsed it to a single "Ext." cell — so the Kobra X showed only the first external slot and the V2's external slot was missing. Reworked `cards.js` to render **every box as its own row with all its slots** (external `id -1` → E1–E4, ACE units → A/B/C/D…), so a Kobra X now shows all 20 slots. Dev tools: `scripts/acu-lan-layout.mjs` + `acu-cloud-layout.mjs` (box/slot dumps).

## Changed
- Extra subnets to scan — unified into **one shared list** across all 5 brand scan modals (Bambu Lab, Snapmaker, Creality, Elegoo, FlashForge) instead of 4 independent per-brand localStorage keys + 1 brand with no UI. List now lives at `users/{uid}.scanExtraSubnets` in Firestore, so a user who declares "scan 192.168.20 too" once sees it honoured by every brand and synced to every signed-in device. localStorage cache layer (`tigertag.scanExtraSubnets`) survives offline / cold-start. A one-time migration on first run merges the 4 legacy keys (`tigertag.{snap,cre,elg,ffg}ScanExtraSubnets`) into the unified store and deletes them. Shared logic lives in `renderer/printers/extra-subnets.js` — chip renderer subscribes to list changes so a prefix added inside one open modal updates the chip list of any other open modal in real time — `renderer/printers/extra-subnets.js` (new), `renderer/printers/{snapmaker,creality,elegoo,flashforge,bambulab}/add-flow.js`, `renderer/inventory.js`.

## Fixed
- FlashForge — Scan du réseau : extra subnets entered by the user were silently wiped at every scan start (`_ffgExtraSubnets = []` in the scan-start handler) — so a user who added `192.168.20` and clicked Scan never actually scanned that subnet. Fixed as part of the unified extra-subnets refactor.
- FlashForge LAN scan finds printers on user-declared extra subnets again, matching the Flutter mobile app. Three changes:
  - **TCP M115 fallback now fires when HTTP `/detail` returns a FlashForge-shaped placeholder** (`{"code":1,"message":"SN is different"}` — what AD5X / 5M / 5M Pro firmwares return to an empty-credentials probe). Previously the renderer required at least one identity field in the HTTP response and `return null`-ed before reaching the TCP fallback, so a perfectly reachable printer on a routed subnet was silently skipped (verified live with `curl` to `:8898/detail` and `nc :8899 ~M115` against the printer on 192.168.20.141). The probe now treats `"code" in flat || "detail" in flat || "message" in flat` as a "weak match" worth enriching with the TCP probe.
  - Per-host HTTP probe timeout for extra subnets is now **900 ms** (was 350 ms — too tight for cross-VLAN routed replies whose RTT routinely exceeds 350 ms).
  - Parallelism on extra subnets is back at **16 probes per batch with no gap** (was 4 + 50 ms gap — dragged a /24 sweep past 25 s).

  Multicast discovery also now matches the Flutter implementation exactly: `setMulticastLoopback(false)`, options applied in the same order (TTL → loopback → addMembership → send), and the "Hello World!" probe is sent twice 50 ms apart to survive a router warming up its IGMP-snoop table — `main.js`, `renderer/printers/flashforge/probe.js`.
- Printer settings panel tutorial pill — was always opening the same tutorial regardless of which model the user picked. The pill now reads the live model name from the picker (`#pbaMpTriggerText.textContent`) at click time and routes to the correct series, instead of being pinned to `defaultModel` at render time — most visible on Bambu Lab where 11 models map to 3 different tutorials (X1/X1E/H2/P2 → X1 series · P1P/P1S → P1 series · A1/A1 mini → A1 series) — `renderer/printers/modal-helpers.js`.
